자연어 ERD 제어 — LangGraph 기반 Plan-and-Execute 아키텍처 설계서

LangChain · LangGraph · OpenAI · UXERManager (Python Proxy 우선)
버전 v0.2 (LangGraph 기반) 작성일 2026-05-30 두뇌 Python Proxy + LangGraph 손발 클라이언트 툴 런타임

1.개요 · 목표

사용자가 자연어로 입력하면, Python 프록시에 호스팅된 LangGraph 상태 그래프가 의도를 분기하고(행동/답변), 계획을 세우고, 클라이언트에 툴 실행을 위임하며, 결과를 관측해 재계획한다. 클라이언트는 ENTITIES/RELATIONS를 쥔 툴 실행 런타임으로서 그래프의 interrupt에 응답한다.

🎯 핵심 목표
  • 자연어 → 의도 파악 (LangGraph 라우팅 노드)
  • 행동 불필요 질의는 툴 없이 토큰 스트리밍 응답
  • 지원 툴 범위 내 계획 수립·실행 (plan-and-execute)
  • 툴 독립 구동 · 병렬 · 순차(출력→입력 전달)
  • 양방향: 프록시 그래프가 클라이언트에 데이터 요청
✅ LangGraph 도입 효과
  • Checkpointer로 세션·재개를 프레임워크가 처리
  • interrupt/Command로 client-in-the-loop 표준화
  • 조건부 엣지로 재계획·분기 선언적 표현
  • astream_events로 토큰·진행 스트리밍 일원화
  • 손수 만든 DAG 엔진·세션 스토어 제거 → 유지보수↓

왜 LangGraph인가

Plan-and-Execute는 본질적으로 상태를 가진 순환 그래프다(계획→실행→관측→재계획→…). 이를 손으로 구현하면 세션 보관·재개·중단/재개·스트리밍·재시도를 모두 직접 만들어야 한다. LangGraph는 바로 이 문제를 위한 프레임워크로, StateGraph·Checkpointer·interrupt·조건부 엣지를 1급 시민으로 제공한다. 특히 LangGraph 공식 Plan-and-Execute 레시피가 본 요구사항과 정확히 일치한다.

⚠️ 본 프로젝트의 특수점
표준 plan-and-execute는 툴을 서버에서 실행한다. 그러나 UXERManager의 ERD 상태(ENTITIES)는 클라이언트 메모리에 있다. 따라서 본 설계는 LangGraph의 interrupt()(human-in-the-loop)를 client-in-the-loop으로 활용한다 — 그래프가 툴 실행 지점에서 멈춰 클라이언트에 실행을 위임하고, 클라이언트가 라이브 상태에서 실행한 결과로 Command(resume=…) 하여 그래프를 재개한다.

그림 1 — 프록시가 LangGraph 그래프를 호스팅(두뇌), 클라이언트는 툴 런타임(손발). interrupt↔resume으로 실행을 왕복 위임한다.

4.LangGraph 상태 그래프

4.1 AgentState

그래프의 모든 노드가 읽고 쓰는 공유 상태. 순차 실행의 "출력→입력 전달"은 이 상태 누적으로 자연스럽게 구현된다.

from typing import Annotated, TypedDict, Literal
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    messages:   Annotated[list, add_messages]   # 대화·툴 메시지 누적
    erd_context: dict          # 클라이언트가 보낸 ERD 요약(엔티티/관계 id·이름)
    route:       Literal["act", "answer"] | None
    plan:        list[dict]     # 남은 스텝 목록
    past_steps:  Annotated[list, lambda a,b: a+b]  # (step, result) 누적
    response:    str | None    # 최종 응답

4.2 노드 · 엣지 (그래프 토폴로지)

from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver   # 운영: SqliteSaver

g = StateGraph(AgentState)
g.add_node("gate",    gate_node)        # 의도 분기
g.add_node("answer",  answer_node)      # 툴없이 직접 답변(스트리밍)
g.add_node("plan",    plan_node)        # 계획 수립(구조화 출력)
g.add_node("execute", execute_node)     # 클라이언트에 툴 실행 위임(interrupt)
g.add_node("replan",  replan_node)      # 관측→재계획/종료 판단
g.add_node("respond", respond_node)     # 최종 요약 응답

g.add_edge(START, "gate")
g.add_conditional_edges("gate",   lambda s: s["route"],
                        {"answer":"answer", "act":"plan"})
g.add_edge("answer", END)
g.add_edge("plan",    "execute")
g.add_edge("execute", "replan")
g.add_conditional_edges("replan", should_continue,
  {"continue":"execute", "revise":"execute", "alternative":"execute",   # 적응형 재계획 §6.4
                         "done":"respond", "escalate":"respond", "abort":"respond"})
g.add_edge("respond", END)

graph = g.compile(checkpointer=MemorySaver())   # thread_id 별 상태 영속
┌──────────────────────────── replan(loop) ──┐ ▼ │ START ─▶ gate ──act──▶ plan ─▶ execute ─▶ replan ──continue───────┘ │ ▲ │ answer interrupt└──done──▶ respond ─▶ END │ (클라 실행) ▼ END (토큰 스트리밍)
그림 2 — StateGraph 토폴로지. gate가 answer/act로 분기, execute↔replan이 plan-and-execute 순환을 이룬다.

6.클라이언트 인 더 루프 (interrupt 기반)

ERD 툴의 실행과 데이터 조회는 모두 클라이언트에서 일어난다. LangGraph의 interrupt()가 이 위임의 단일 메커니즘이다 — v0.1의 need_data/resolve와 DAG 실행을 하나로 통합한다.

6.1 interrupt / resume 프로토콜

# execute_node — 서버(프록시) 측
from langgraph.types import interrupt

def execute_node(state: AgentState):
    batch = next_parallel_batch(state["plan"])      # 의존성 없는 스텝 묶음
    tool_calls = [{"id":s["id"], "tool":s["tool"], "args":s["args"]} for s in batch]
    # ↓ 그래프 일시정지. 페이로드가 클라이언트로 전달됨
    results = interrupt({"type":"tool_calls", "calls": tool_calls})
    # ↓ 클라가 Command(resume=results) 하면 여기서 재개
    return {"past_steps": [(c, r) for c,r in zip(tool_calls, results)],
            "messages": to_tool_messages(tool_calls, results)}
// 클라이언트 — interrupt 수신 → 툴 실행 → resume
async function onInterrupt(payload, threadId) {
  const { calls } = payload;
  // 병렬 실행 (의존성 없는 배치)
  const results = await Promise.allSettled(
    calls.map(c => registry[c.tool].run(ctx.draft, c.args))
  ).then(rs => rs.map(toResult));
  // resume — SSE 스트림 재개
  await postSSE('/agent/resume', { threadId, resume: results });
}
왜 이 방식이 우아한가
thread_id로 식별되는 그래프 상태는 Checkpointer에 저장돼 있으므로, HTTP 왕복 사이에 프록시가 대화 맥락을 직접 보관할 필요가 없다. interrupt는 한 지점에서 데이터 조회든 상태 변경 툴이든 동일하게 클라이언트로 위임한다 — v0.1의 두 메커니즘이 하나로 합쳐진다.

6.2 병렬 · 순차 · 데이터 전달 (LangGraph 표현)

요구LangGraph 구현
병렬① execute 한 번의 interrupt에 독립 툴콜 배치를 실어 보내 클라가 Promise.allSettled로 동시 실행. ② 또는 Send API로 노드를 팬아웃해 슈퍼스텝에서 병렬 처리.
순차execute→replan→execute 순환. 각 라운드 결과가 past_steps에 누적되고 다음 plan/execute가 이를 참조.
출력→입력 전달선행 툴 결과가 State(past_steps)에 들어가므로, 다음 스텝의 인자를 만들 때 LLM/바인더가 실제 값(예: 새 entityId)을 사용. v0.1의 ${n1.id} 토큰 치환을 State 누적이 대체.
# Send 로 병렬 팬아웃 (대안)
from langgraph.types import Send
def fan_out(state):
    return [Send("execute_one", {"step": s}) for s in independent_steps(state)]
g.add_conditional_edges("plan", fan_out)   # 독립 스텝들이 동시 슈퍼스텝으로 실행

6.3 양방향 데이터 해소 (계획 중 조회)

"회원 테이블에 이메일 추가해줘" → 계획하려면 회원 테이블의 현재 컬럼이 필요하다. 이는 plan/execute 노드가 읽기 툴(describe_table)을 먼저 interrupt로 요청해 해결한다. 별도 프로토콜이 아니라 같은 interrupt 메커니즘이다.

그림 3 — 양방향 데이터 해소 시퀀스. 데이터 조회와 변경 실행이 모두 interrupt(노랑, 클라 위임)↔resume(초록, 재개)으로 통합되며, 그 사이 그래프는 일시정지되고 Checkpointer가 thread_id별 상태를 보존한다.

6.4 적응형 재계획 — 실패·이탈 시 대체 수단

계획대로 진행하다 예기치 않은 상황(툴이 오류를 내거나, 결과가 기대와 다른 방향)이 생겨 그대로 진행하는 게 무의미해지면, replan 노드가 이를 감지해 대체 수단으로 계획을 수정한다. 이것이 단순 "계획 실행"과 plan-and-execute의 결정적 차이다 — 실행은 항상 실행 → 결과 평가 → 적응형 재계획의 순환 안에서 일어난다.

두 가지 실패 유형

유형예시감지
경성 실패 (hard)툴 예외·HTTP 오류·인자 스키마 검증 실패·DB 연결 실패예외/error 결과가 past_steps에 기록됨 — 명시적
연성 실패 · 이탈 (soft)툴은 성공했지만 빈 결과·0건, 잘못된 방향, 후속 스텝의 전제 위반, 여러 라운드째 진전 없음결과 검증 + 목표 대비 LLM 자기평가 — 추론 필요

"기대와 다름"을 어떻게 아는가 — 평가 신호

  • 구조적 검증: 결과 스키마·빈값·카운트(예: find_tables 0건, describe_table 미존재).
  • 스텝 성공조건(postcondition): plan 수립 시 각 스텝에 expectation을 부여 → replan이 실제 결과와 대조.
  • LLM 자기평가(reflexion): "원래 목표 대비 현재 진척이 유효한가?"를 replan이 판단.
  • 진전 없음 감지: 동일 오류 반복·replan_count 상한 → 같은 길로 더 가지 않음.
그림 4 — replan의 5갈래 분기. 정상이면 continue, 고칠 수 있으면 revise, 접근 자체가 무의미하면 alternative(대체 수단), 사람이 정해야 하면 escalate, 안되면 abort. revise·alternative는 execute로 되돌아 순환을 잇는다.

라우터 · 상태 확장

from typing import Literal

def should_continue(state) -> Literal["continue","revise","alternative","escalate","abort"]:
    last = state["past_steps"][-1]
    if last.ok and meets_expectation(last):    return "continue" if state["plan"] else "done"
    if state["replan_count"] >= MAX_REPLAN:    return "escalate"   # 진전 없음 → 사람에게
    if recoverable(last):                    return "revise"      # 인자·순서만 고치면 됨
    if has_alternative(state):               return "alternative" # 대체 수단 있음
    return "abort"                            # 안전 종료

# AgentState 확장 필드
#   past_steps: [{step, ok, error, result}]   ← 오류·결과 동반
#   plan[*]:    {tool, args, expectation, alternatives:[...]}  ← 기대·대체안
#   replan_count: int                          ← 무한루프 방지

g.add_conditional_edges("replan", should_continue, {
    "continue":"execute", "revise":"execute", "alternative":"execute",
    "done":"respond", "escalate":"respond", "abort":"respond",
})
🛡️ 무의미한 진행을 막는 안전장치
  • 드래프트 격리: 모든 변경은 클라이언트 드래프트에만 → 실패해도 실제 ERD는 안전. 대체 시도는 드래프트를 실패 직전으로 롤백한 뒤 진행.
  • 시도 상한: replan_count + 그래프 recursion_limit → 같은 실패를 무한 반복하지 않고 escalate로 사람에게 넘긴다.
  • 되돌리기 어려운 작업(run_sql DML/DDL, delete_*)은 대체 시도 전 반드시 승인 interrupt를 거친다.
  • 관측: 모든 분기(revise/alternative/abort)와 사유는 astream으로 사용자에게 표시되고 감사 로그에 남는다.

6.5 휴먼 인 더 루프 (HITL) — 승인 · 편집 · 질의

가능하다 — 그리고 토대가 이미 깔려 있다. LangGraph의 interrupt()는 본래 human-in-the-loop 프리미티브다. §6의 client-in-the-loop은 같은 메커니즘을 자동 툴 실행에 쓴 것이고, 여기에 사람의 판단을 끼우면 그대로 HITL이 된다 — 그래프는 interrupt로 멈추고, 사람이 Agent 패널에서 결정하면 그 값으로 Command(resume=…) 하여 재개한다.

두 가지 "인 더 루프" — 같은 메커니즘, 다른 응답자

구분interrupt 대상응답 주체
client-in-the-loop클라 런타임에 툴 실행 위임(§6.1)코드 — 자동
human-in-the-loop사람에게 결정·입력 요청사람 — Agent 패널 UI

둘 다 동일한 interrupt ↔ Command(resume) + Checkpointer 위에서 동작하므로, HITL은 새 인프라 없이 노드 한두 개 추가로 얹힌다.

HITL 패턴 ↔ 본 설계 적용

그림 5 — HITL 승인 흐름. approval_gate가 interrupt로 멈추면(그래프 일시정지) 사람이 패널에서 결정하고, resume된 decision에 따라 approve→execute · edit→plan 수정 · reject→replan으로 분기한다. 대기 중 상태는 Checkpointer가 보존하므로 비동기 승인도 가능하다.

승인 게이트 노드 (코드 스케치)

from langgraph.types import interrupt, Command

def approval_gate(state):
    # 위험·비가역 스텝 앞에서 사람의 결정을 요청 (그래프 일시정지)
    decision = interrupt({
        "type": "approval",
        "summary": describe(state["pending_step"]),   # "주문 테이블 삭제" 등
        "risk": "irreversible",
        "options": ["approve", "edit", "reject"],
    })
    # ↓ 사람이 Agent 패널에서 선택 → Command(resume=decision) 으로 재개
    if decision["action"] == "reject": return Command(goto="replan")      # 거부 → 대체안 모색
    if decision["action"] == "edit":   return {"plan": apply_edits(state["plan"], decision["patch"])}
    return {}   # approve → 그대로 진행
// 클라이언트 — 승인 카드에서 [실행]/[수정]/[거부] 선택 → resume
await postSSE('/agent/resume', { threadId, resume: { action: 'approve' } });
// 편집: { action:'edit', patch:{ stepId:'n3', args:{ type:'VARCHAR(255)' } } }

승인 정책 (설정 가능)

정책동작
per-tooldelete_*·run_sql(DML/DDL) 등 비가역 툴은 항상 승인 필요
per-plan기본 — 전체 계획 1회 승인 / 엄격 모드 — 스텝별 승인
자동 승인읽기 툴·레이아웃 등 비가역성 낮은 작업은 무승인(화이트리스트)
⏸ Checkpointer 시너지 — 비동기 HITL
사람이 즉시 결정하지 않아도 그래프 상태는 thread_id로 보존된다(§9). 따라서 승인 요청을 띄운 뒤 자리를 비웠다가 나중에 돌아와 이어서 승인할 수 있다(SqliteSaver면 앱 재시작 후에도). 모든 승인·거부·편집 결정은 감사 로그에 기록된다. 기본 UX는 "계획 1회 승인", 위험 작업은 추가 승인이며, 거부 시 replan이 축소·대체안을 제시한다.

7.툴 카탈로그 — 이중 런타임

툴은 클라이언트와 프록시 양쪽에 존재한다. 모든 툴은 LangChain @tool 스키마로 정의돼 LLM에 동일하게 노출되지만, 각 툴은 location 속성(client | proxy)을 가지며 execute 노드가 이 값으로 실행 위치를 라우팅한다.

  • 클라이언트 툴 — ERD 상태(ENTITIES) 조작·조회. interrupt↔resume으로 위임(§6).
  • 프록시 툴 — 실제 DB 접속·연계 작업, 그리고 향후 MCP 서버 연계. 그래프 안에서 ToolNode로 직접 실행(클라 왕복 불필요).
그림 6 — 이중 툴 런타임. execute 노드가 툴의 location으로 분기: 클라이언트 툴은 interrupt로 위임, 프록시 툴(DB·MCP)은 그래프 안 ToolNode로 직접 실행한다.

7.1 클라이언트 툴 — ERD 조작·조회 read write

프록시의 함수 본문은 실제 로직 대신 interrupt로 위임만 하는 스텁이다(LLM엔 시그니처만 의미).

@tool
def create_entity(logicalName: str, physicalName: str, attrs: list) -> dict:
    """새 엔티티(테이블)를 생성한다. 반환: {entityId}"""
    raise ClientToolDelegation()      # location="client" → execute 노드가 interrupt로 위임

7.2 프록시 툴 — DB 연계 · MCP external

프록시 툴은 본문에 실제 로직을 담아 그래프 안에서 직접 실행한다. ToolNode 또는 execute 노드가 종류로 구분해 호출한다.

@tool
async def run_sql(sql: str) -> dict:
    """연결된 DB에 SQL을 실행한다(읽기 기본). 기존 /execute 로직 재사용."""
    return await db_execute(sql)     # location="proxy" → 클라 왕복 없이 서버 실행

# 향후: MCP 서버 툴을 LangChain 툴로 편입
from langchain_mcp_adapters.client import MultiServerMCPClient
mcp = MultiServerMCPClient({"std-mcp": {"transport":"sse", "url": ...}})
proxy_tools = [run_sql, fetch_db_schema, *await mcp.get_tools()]
툴구현 / 연계
run_sql기존 /execute — DB에 SQL 실행(읽기 기본, DML/DDL은 승인)
fetch_db_schema기존 /schema — 운영 DB의 테이블·뷰·FK 조회 → 역공학 입력
test_connection / list_profiles기존 /config — 접속정보·프로파일 (DB 접속 연계)
확장 MCP 서버 툴향후 — langchain-mcp-adapters로 MCP 서버의 툴을 프록시 LangChain 툴로 자동 편입. 표준 외부 시스템(사내 API·검색·파일·표준용어 등) 연계
실행 위치 라우팅 규칙
  • location="client" (ERD read·write) → execute 노드가 interrupt로 클라이언트 위임. 라이브 상태 일관성·원자적 커밋 보장.
  • location="proxy" (DB 연계·MCP) → 그래프 안 ToolNode로 직접 실행. 네트워크·키·세션이 이미 프록시에 있어 왕복 불필요.
  • 그래프는 하나의 plan에서 두 종류 툴을 혼합할 수 있다(예: 프록시 fetch_db_schema → 클라 create_entity×N → 클라 auto_layout).

from langchain_openai import ChatOpenAI
from pydantic import BaseModel

llm      = ChatOpenAI(model="gpt-4o",      api_key=load_openai_key(), temperature=0)

# /agent/stream — astream으로 토큰·interrupt를 SSE 중계
@router.post("/stream")
async def stream(body: TurnBody):
    cfg = {"configurable": {"thread_id": body.threadId}}
    async def gen():
        async for ev in graph.astream(
                {"messages":[("user",body.query)], "erd_context":body.context},
                cfg, stream_mode=["messages","updates"]):
            kind, data = ev
            if kind=="messages":                       # answer/respond 토큰
                yield sse("token", {"t": data[0].content})
            elif "__interrupt__" in data:               # 클라 실행 위임
                yield sse("interrupt", data["__interrupt__"][0].value)
                return                                  # resume 호출 대기
        yield sse("done", {})
    return StreamingResponse(gen(), media_type="text/event-stream")

9.Checkpointer · 세션

v0.1의 손수 만든 세션 스토어를 Checkpointer가 대체한다. 대화·재개 단위는 thread_id다.

관심사LangGraph 처리
세션 식별thread_id = 클라가 턴 시작 시 발급(예: sess_8fk2). interrupt/resume이 같은 thread를 공유
상태 영속개발: MemorySaver(인메모리) / 운영: SqliteSaver(~/.uxermanager/agent.db) — 앱 재시작에도 진행 중 턴 복구

TTL · 정리오래된 thread는 배경 작업으로 정리(또는 Sqlite에 보존 후 수동 삭제)

⚠️ 상태 드리프트 방어
interrupt↔resume 사이 사용자가 ERD를 직접 수정할 수 있다. erd_context에 버전 토큰을 포함하고, resume 결과에도 버전을 실어 보내 그래프가 불일치를 감지하면 describe_table로 재조회하도록 한다.

13.보안

  • 키 격리: OpenAI 키는 프록시 키스토어(AES-256-GCM, ~/.uxermanager)에만. ChatOpenAI가 서버에서 사용 — 브라우저 노출 없음.
  • CORS 화이트리스트: 기존 ALLOWED_ORIGINS 유지.
  • SQL 가드: run_sql은 기본 읽기 전용 화이트리스트, DML/DDL은 승인 interrupt + 감사 로그(write_audit_log).
  • 프롬프트 인젝션 완화: LLM은 상태를 직접 못 바꾼다 — 모든 변경은 카탈로그 툴 + 클라이언트 드래프트 + 승인 게이트를 거친다.
  • 관측·감사: 그래프 단계·툴콜·SQL은 astream 이벤트로 표시되고 감사 로그에 기록.
  • 로컬 only: 그래프·Checkpointer는 127.0.0.1 프록시 안에서만 동작.
