# 파일 ▸ 가져오기 — 분석 보고서

## 1. 개요

`js/import.js`는 JSON 불러오기, 전체 백업 복원, DDL→ERD 역파싱, AI 스키마 생성, 다이어그램 선택/덮어쓰기/새탭 모달, 파일 input 핸들러를 담당한다. 전반적으로 `escHtml`로 XSS를 방어하고, `render`/`saveState`/`renderEntityTree` 갱신과 `migrateEntity` 마이그레이션을 대부분의 경로에서 일관되게 호출하며, 전체 백업 그룹(`_BK_GROUPS`·`_doExportWithGroups`·`_doImportWithGroups`)도 대칭적으로 구성되어 영속성 무결성은 양호하다. 다만 세 가지 버그가 확인되었다: 레거시 `entities` 직접 포함 형식 import 경로에서 `migrateEntity` 호출이 누락되어 구버전 데이터가 깨지고, 다이어그램 선택 모달의 배경 클릭 닫기가 `overlayClose`에 케이스가 없어 동작하지 않으며, DDL 가져오기 시 엔티티 ID 충돌로 이름이 바뀌면 `relations` 배열이 재매핑되지 않아 관계가 잘못 연결되는 문제가 있다.

## 2. 발견된 오류 및 수정 결과

### [medium] 레거시 JSON 불러오기 경로에서 migrateEntity 누락 → 구버전 데이터 깨짐

- **상태**: 수정완료
- **증상**: 최상위에 `entities` 배열을 가진 레거시 형식 JSON을 '가져오기'로 불러오면 엔티티가 마이그레이션 없이 그대로 새 다이어그램에 들어간다. 구버전 엔티티/속성은 `logicalName`·`physicalName` 대신 `name` 필드를 쓰고 `ref`/`notNull`/`unique`/`defaultValue`/`description`/`indexes`/`isView` 같은 필드가 없어, import 직후 워크스페이스에서 엔티티/속성 이름이 빈 값(또는 id 폴백)으로 표시되고 속성 편집·DDL 생성 시 누락 필드로 인한 오작동이 발생한다. 동일 파일의 다른 모든 import 경로(`_doImportWithGroups` L37, `_applyImportDiagram` L192, `_addImportDiagramAsNew` L211)와 `state.js`의 `loadState` 레거시 분기(L153)는 모두 `migrateEntity`를 호출하지만, 이 레거시 분기만 누락되어 일관성이 깨진다.
- **원인**: `createEmptyDiagram`으로 만든 `d`에 `data.entities`를 `migrateEntity` 없이 직접 대입(`d.entities = data.entities`). `loadDiagramIntoWorkspace`와 `flushCurrentState`는 마이그레이션을 수행하지 않고 깊은 복제만 하므로, 다음 `loadState`/`restoreFromSnapshot`이 호출되기 전까지 미마이그레이션 상태가 그대로 화면에 노출된다.
- **수정 내용**: 레거시 JSON 경로에서 `d.entities`에 `migrateEntity` 매핑을 추가하여 구버전 데이터 깨짐 방지
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\import.js`

### [medium] 다이어그램 선택 모달(importDiagSelectOverlay) 배경 클릭 닫기 미동작

- **상태**: 수정완료
- **증상**: `index.html`(L528)에서 `importDiagSelectOverlay`는 `onmousedown="overlayClose(event,'importDiagSelectOverlay')"`로 배경 클릭 닫기가 연결돼 있으나, `overlayClose` 함수에는 `importDiagSelectOverlay`에 대한 케이스가 없고 generic 닫기 처리도 없다. 따라서 배경(딤 영역)을 클릭해도 모달이 닫히지 않아 다른 모든 모달과 동작이 불일치하고, `_importFileDiagrams` 임시 배열도 정리되지 않는다(취소/덮어쓰기/새탭 버튼으로만 닫힘). 같은 계열의 `ddlImportOverlay`·`aiSchemaOverlay`는 `overlayCloseExtra`를 통해 정상적으로 닫힌다.
- **원인**: `overlayClose`의 `if` 분기 목록에 `'importDiagSelectOverlay'` → `closeImportDiagSelectModal()` 매핑이 빠져 있다. 이 overlay는 `overlayCloseExtra`가 아니라 `overlayClose`에 연결되어 있는데 `overlayClose`에도 해당 id 처리가 없어 배경 클릭이 무동작(no-op)이 된다.
- **수정 내용**: `overlayClose` 함수에 `importDiagSelectOverlay` 배경 클릭 시 `closeImportDiagSelectModal()` 호출 분기 추가
- **파일**: `js/ui.js`

### [medium] DDL 가져오기 ID 충돌 시 relations 미재매핑 → 관계 오연결/누락

- **상태**: 수정완료
- **증상**: `applyDDLImport`는 기존 ENTITIES와 id가 충돌하면 `e.id`를 `e.id+'_2'` 등으로 변경(L367~379)하지만, `parseDDL`이 만든 `relations` 배열의 `from`/`to`(원래 파싱된 id 기준)는 전혀 갱신하지 않는다. 같은 DDL을 두 번 가져오거나 기존 엔티티와 테이블명이 겹치는 DDL을 가져오면, 새로 추가된 테이블은 `dept_2`/`users_2`로 들어가는데 관계는 여전히 원래 id(`dept`/`users`)를 가리킨다. 그 결과 (a) 새 테이블에는 관계가 그려지지 않고, (b) 관계가 우연히 먼저 들어와 있던 옛 엔티티에 잘못 연결되거나 고아 관계가 된다. 또한 L373~378의 `a.ref.entity` 재매핑도 같은 루프 안에서 먼저 이름이 바뀐 엔티티를 원래 id로 찾으려 해 실패하여 FK ref가 옛 엔티티를 가리킨다.
- **원인**: ID 디듀프 시 `oldId`→`newId` 매핑을 만들어 `relations.from`/`to` 및 `a.ref.entity`에 적용하지 않고, 엔티티 객체의 `id`만 바꿨다. `relations.forEach` 푸시(L386~390)에는 재매핑 로직이 전혀 없다.
- **수정 내용**: DDL 가져오기 시 `idRemap` 테이블로 ID 충돌 리네이밍을 추적하여 FK ref 및 `relations` `from`/`to`를 확정 ID로 재매핑
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\import.js`

## 4. 추천 사항

- **DDL 가져오기 dialect 자동 감지 또는 선택 UI**: 현재 `parseDDL`은 MySQL/PostgreSQL/Oracle을 단일 정규식으로 처리한다. 백틱/대괄호/큰따옴표 식별자, `AUTO_INCREMENT` vs `SERIAL` vs `GENERATED AS IDENTITY` 등 방언별 단서를 감지해 `autoIncrement`/타입 정규화를 더 정확히 채우고, 모달에 방언 선택 드롭다운(또는 자동 감지 표시)을 추가하면 round-trip(생성→가져오기) 정확도가 높아진다.

- **DDL 가져오기에 컬럼별 NOT NULL/UNIQUE/DEFAULT/AUTO_INCREMENT 완전 추출**: 현재 파서는 `notNull`과 `pk`/`fk`만 충실히 채우고 `unique`/`defaultValue`/`autoIncrement`는 사실상 기본값(`false`/`''`)으로 둔다. 컬럼 라인에서 `UNIQUE`, `DEFAULT <expr>`, `AUTO_INCREMENT`/`SERIAL`/`IDENTITY`를 파싱해 attr에 반영하면 `export.js`의 `buildDDL`과 무손실 왕복이 가능해진다.

- **import 결과를 단일 undo 트랜잭션으로 묶고 취소 안내**: 전체 백업 복원은 `undoStack`을 `[mainSnap]`으로 리셋해 이전 상태로 되돌릴 수 없다. 복원 전 스냅샷을 1단계 보관하거나 복원 직후 '되돌리려면 Ctrl+Z' 토스트를 제공하면 실수로 인한 데이터 유실 위험을 줄일 수 있다.

- **AI 스키마 생성 결과 미리보기 단계 추가**: DDL 가져오기에는 `previewDDLImport`가 있으나 AI 스키마는 응답을 곧바로 `applyAISchema`로 반영한다. 파싱 결과를 먼저 미리보기로 보여주고 사용자가 확인 후 적용하도록 하면 잘못된 LLM 출력의 즉시 반영을 막을 수 있다.
