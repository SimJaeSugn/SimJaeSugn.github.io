# 편집 ▸ 엔티티 — 분석 보고서

## 1. 개요

`js/entities.js`는 엔티티 모달(`entOverlay`)을 중심으로 엔티티 CRUD, 속성 행(추가·삭제·드래그 이동·PK/FK/일반 토글·FK 참조 드롭다운), 인덱스, 색상 태그, 예상 행수(볼륨), 속성 추출, 템플릿 적용, 복사/붙여넣기, 다이어그램 간 복사, FK 컬럼 자동생성을 담당한다. `index.html`의 모든 `onclick` 핸들러와 모달 DOM id는 정상 연결되어 있으며, 단축키(`addEnt=N`, copy/paste/dup/del)도 `main.js`·`shortcuts.js`·메뉴바와 동기화되어 있다. `dbType`/`rowCount`/`colorTag`/`indexes`/`attrs`는 `diagram` 객체에 저장되어 `saveState` 직렬화 및 전체 백업에 포함된다. 전반적으로 견고하나, 속성 추출 후 좌측 엔티티 트리 미갱신과 FK 참조 드롭다운의 엔티티/속성명 비이스케이프 삽입 두 가지 실제 버그가 확인되었다.

## 2. 발견된 오류 및 수정 결과

### [medium] 속성 추출(doExtractEntity) 후 좌측 엔티티 트리(renderEntityTree) 미갱신 → 화면-상태 불일치

- **상태**: 수정완료
- **증상**: 엔티티 편집 모달에서 '속성 추출'을 실행하면 새 엔티티가 ENTITIES에 추가되고 캔버스에는 `render()`로 그려지지만, 좌측 엔티티 트리 패널은 `renderEntityTree()`를 호출하지 않아 새로 생성된 엔티티가 트리에 즉시 나타나지 않는다. 또한 `keep=false`일 때 원본 엔티티의 속성이 줄어든 것도 트리 attr count에 반영되지 않는다. 다른 엔티티 생성/삭제 경로(`saveEntity` 325행, `deleteEntity` 335행, `pasteEntity` 557행)는 모두 `renderEntityTree()`를 호출하는데 이 경로만 누락되어 트리가 다음 트리 갱신 동작 전까지 stale 상태로 남는다.
- **원인**: `doExtractEntity()`의 마무리 블록에서 `render()`와 `saveState()`는 호출하지만 `renderEntityTree()`를 호출하지 않음(점검 버그클래스 3: render/renderEntityTree 갱신 누락).
- **수정 내용**: `doExtractEntity` 완료 후 `render`/`saveState` 다음에 `renderEntityTree()` 호출을 추가해 좌측 엔티티 트리가 즉시 갱신되도록 수정.
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\entities.js`

### [low] FK 참조 엔티티 드롭다운(populateRefEntitySelect)에 엔티티명 비이스케이프 삽입 → 옵션 markup 깨짐/잘못된 ref

- **상태**: 수정완료
- **증상**: FK 속성 행의 '참조 엔티티' 드롭다운을 채울 때 `e.logicalName`/`e.physicalName`/`e.id`를 `escHtml` 없이 `innerHTML` 옵션 텍스트와 `value` 속성에 직접 삽입한다. 엔티티 논리명에 `<`, `>`, `&` 또는 따옴표가 포함되면 option 텍스트/값이 깨져 잘못된 FK 참조가 선택되거나 드롭다운이 손상된다(XSS 가능성 포함). 프로젝트 표준 `escHtml`(`ui.js`)과 `renderEntityTree` 등 다른 곳은 이미 `escHtml`로 이스케이프하는데 이 함수만 누락되어 일관성이 깨진다.
- **원인**: 사용자 입력(엔티티명)을 `escHtml` 없이 `innerHTML` 템플릿 리터럴에 삽입(점검 버그클래스 11). `value="${e.id}"`와 옵션 텍스트 `${e.logicalName||e.physicalName||e.id}` 모두 비이스케이프.
- **수정 내용**: `populateRefEntitySelect`에서 엔티티명과 ID를 `escHtml()`로 감싸 XSS 및 마크업 깨짐 방지.
- **파일**: `js/entities.js`

### [medium] FK 참조 속성 드롭다운(populateRefAttrSelect)에 속성명 비이스케이프 삽입 → 옵션 깨짐/잘못된 ref.attr

- **상태**: 수정완료
- **증상**: FK 속성 행의 '참조 컬럼' 드롭다운을 채울 때 속성 물리명/논리명(`pn`, `a.logicalName`)을 `escHtml` 없이 option의 `value`와 텍스트에 직접 삽입한다. 속성명에 특수문자(`<`, `>`, `&`, 따옴표)가 들어가면 `value`가 깨져 저장 시 `result.ref.attr`이 잘못 들어가거나 옵션 markup이 손상된다. `e2`와 동일한 이스케이프 누락 패턴.
- **원인**: 사용자 입력(속성명)을 `escHtml` 없이 `innerHTML` 옵션 `value`/텍스트에 삽입(점검 버그클래스 11).
- **수정 내용**: `populateRefAttrSelect`의 option `value`와 표시 텍스트에 `escHtml()`을 적용해 속성명의 특수문자로 인한 HTML 깨짐 및 잘못된 `ref.attr` 저장 방지.
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\entities.js`

## 4. 추천 사항

- **FK 참조 무결성 단방향 정합성 점검 유틸 추가**: 엔티티/속성 삭제·이름변경 시 다른 엔티티의 `attr.ref`(entity/attr)와 RELATIONS의 `from`/`to`가 실제 존재하는 엔티티·속성을 가리키는지 일괄 검증·복구하는 함수를 도입하면, 편집·복사·추출 과정에서 발생하는 dangling FK ref/orphan 관계선을 자동 정리할 수 있다.
- **속성 행 입력 시 물리명 자동 대문자/스네이크 변환 옵션**: `autoAddFkColumn`은 물리명을 `toUpperCase()+'_ID'`로 만드는데, 수동 입력 속성과 표기 규칙이 어긋날 수 있다. 모달에서 논리→물리명 자동 변환(대문자 스네이크) 헬퍼를 제공하면 일관성과 입력 편의가 좋아진다.
- **붙여넣기/복제 시 FK ref 선택적 재매핑 옵션 제공**: `pasteEntity`·`confirmCopyToDiag`·`main.js`의 dup는 모두 `copy.attrs`의 `ref`를 null로 만든다. 그룹 내부에서 FK 대상이 함께 복사된 경우 idMap을 이용해 ref도 재매핑하는 옵션을 두면, 복사 후 FK가 끊기는 사용성 불편을 줄일 수 있다.
- **엔티티 모달 폼 검증 강화 및 중복 속성명 경고**: 동일 물리명 속성, PK 0개/복수 PK, FK인데 ref 미지정 등 흔한 설계 오류를 저장 시 경고하면 DDL 생성·정규화 진단 단계의 오류를 사전에 차단할 수 있다.
