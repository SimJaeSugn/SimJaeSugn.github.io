# 편집 ▸ 관계 — 분석 보고서

## 1. 개요

'편집 ▸ 관계' 메뉴(js/relations.js)는 관계 추가/편집/삭제, 카디널리티(1:1/1:N/N:M), 선 형태(직선/곡선)·패턴(실선/점선)·색상, 관계명, FK 컬럼 자동 추가를 relOverlay 모달로 처리한다. 단축키 'R'(addRel)은 main.js/shortcuts.js/메뉴/단축키 모달 전반에 일관되게 동기화되어 있고, 관계 데이터는 flushCurrentState로 diagrams[].relations에 깊은 복사되어 undo/redo·localStorage·전체백업 모두에 포함되므로 영속성 무결성은 이상 없다. 레거시 lineStyle='curved' 데이터의 pathStyle 마이그레이션도 모달/속성패널 양쪽에서 올바르게 처리된다. 다만 엔티티 셀렉트 옵션 비이스케이프, FK 자동 추가 후 엔티티 트리 미갱신, 끊어진 엔티티 참조 관계 삭제 시 예외라는 세 가지 항목이 검토되었다.

## 2. 발견된 오류 및 수정 결과

### [medium] 관계 모달 엔티티 셀렉트 옵션을 escHtml 없이 innerHTML로 삽입 (깨짐/마크업 주입)

- **상태**: 수정완료
- **증상**: 관계 추가/편집 모달의 시작·끝 엔티티 드롭다운을 채울 때 entDisplayName(e)(엔티티 논리명/물리명)과 e.id를 그대로 innerHTML 템플릿에 삽입한다. 엔티티 논리명은 entLogical 입력에서 자유 텍스트로 받으며 <, >, & 등에 대한 필터가 없다(entities.js의 entLogical 값 저장 경로 확인). 따라서 이름에 < 또는 " 등이 포함되면 option 태그가 깨져 드롭다운에 엔티티가 누락되거나 임의 마크업이 주입된다. 코드베이스의 다른 동일 패턴(join_explorer.js:45의 <option>...escHtml(...))은 모두 escHtml로 이스케이프하고 있어, 이 위치만 누락되어 있다.
- **원인**: ENTITIES.map(e => `<option value="${e.id}" ...>${entDisplayName(e)} (${e.id})</option>`) 에서 사용자 제어 문자열(엔티티 이름·id)을 escHtml() 없이 innerHTML에 직접 보간함.
- **수정 내용**: populateEntitySelects에서 e.id와 entDisplayName(e) 값을 escHtml()로 감싸 innerHTML 삽입 시 XSS/깨짐 방지
- **파일**: js/relations.js

### [low] FK 자동 추가 후 renderEntityTree() 누락으로 좌측 엔티티 트리 미갱신 (화면-상태 불일치)

- **상태**: 수정완료
- **증상**: 'FK 컬럼 자동 추가' 체크 시 saveRelation()이 autoAddFkColumn(from,to,card)을 호출해 끝 엔티티의 attrs 배열에 FK 컬럼을 push한다. 그러나 마지막에 render()(캔버스 재그리기)와 saveState()만 호출하고 renderEntityTree()는 호출하지 않는다. render()/renderNow()는 캔버스만 다시 그리고 좌측 엔티티 트리(diagrams.js의 attrCount·컬럼 목록)는 갱신하지 않으므로, 새로 추가된 FK 컬럼과 늘어난 속성 개수가 다른 동작이 트리를 다시 그릴 때까지 화면에 반영되지 않는다. 동일하게 엔티티 attrs를 변경하는 saveEntity()는 closeEntModal(); render(); saveState(); renderEntityTree(); 로 트리를 명시적으로 갱신한다(entities.js).
- **원인**: 엔티티 attrs를 변경하는 경로(autoAddFkColumn)를 가졌음에도 saveRelation 종료부가 render(); saveState(); 만 호출하고 renderEntityTree()를 누락함.
- **수정 내용**: saveRelation() 완료 후 renderEntityTree() 호출을 추가해 FK 자동 추가 시 좌측 엔티티 트리도 즉시 갱신
- **파일**: js/relations.js

### [low] 끊어진 엔티티 참조 관계 삭제 시 entDisplayName(undefined) 예외로 삭제 확인창 실패

- **상태**: 오류아님
- **증상**: deleteCurrentRelation()은 entDisplayName(em[editingRelation.from]) || editingRelation.from 형태로 엔티티 이름을 구한다. 작성자는 '엔티티가 없으면 id로 폴백'하려는 의도로 || 폴백을 두었으나, em[editingRelation.from]가 undefined인 경우 entDisplayName(undefined)이 e.logicalName 접근에서 TypeError(Cannot read properties of undefined)를 먼저 던져 || 폴백이 동작하지 못한다(entDisplayName은 state.js에서 인자 null 체크가 없음). 로컬 엔티티 삭제 시에는 deleteEntity가 관련 관계를 정리하므로 발생하지 않지만, 가져오기/공유/병합으로 들어온 dangling 참조(존재하지 않는 엔티티를 가리키는 관계)의 관계 편집→삭제 시 예외가 발생해 삭제 확인창이 뜨지 않고 작업이 조용히 중단된다.
- **원인**: entDisplayName()이 null/undefined 인자를 방어하지 않는데, em[...] 조회 결과가 undefined일 수 있는 dangling 참조 상황에서 그대로 전달하여 || 폴백 이전에 역참조 예외가 발생함.
- **수정 내용**: 앵커는 js/relations.js:94-96에 원문 그대로·유일하게 존재하며, entDisplayName(state.js:204-208)이 e.logicalName 등을 인자 null 체크 없이 역참조하므로 entDisplayName(undefined) 호출 시 TypeError가 발생해 || 폴백이 동작하지 못한다는 기술적 전제는 사실이다. 그러나 deleteCurrentRelation()이 실제로 dangling(존재하지 않는 엔티티를 가리키는) 관계를 받을 경로가 없어 거짓양성이다. 근거: (1) getRelationPath(canvas.js:598-601)는 한쪽 엔티티라도 없으면 null을 반환한다. (2) drawRelations(1143-1145)는 dangling 관계를 아예 그리지 않는다. (3) editingRelation을 설정하는 모든 경로가 getRelationPath가 non-null을 반환해야 통과하는 히트테스트 결과에서만 채워진다. 따라서 dangling 관계는 렌더·히트테스트·선택·편집모달 진입 자체가 불가능하므로 deleteCurrentRelation()에 missing endpoint를 가진 editingRelation이 도달할 수 없다. 제안된 방어적 하드닝 자체는 올바르고 무해하지만 실제 버그를 고치는 것이 아니므로 자동수정 보류로 판정한다.
- **파일**: js/relations.js

## 3. 개선점 (자동수정 안 함 — 권장)

| 우선순위 | 제목 | 대상 파일 |
|---------|------|----------|
| low | 관계 편집 시 변경 후 중복 검사 미수행 — saveRelation()의 중복 검사는 신규 추가에만 적용되며, 기존 관계를 편집해 from/to를 이미 존재하는 조합으로 바꿔도 중복 검사가 동작하지 않는다. 편집 시에도 자기 자신을 제외한 동일 (from,to) 중복을 검사하면 일관성이 높아진다. | js/relations.js |
| low | 신규/편집 모두 한 방향만 중복으로 간주 — 중복 검사는 r.from===from && r.to===to 한 방향만 보는 반면, saveEntity()의 FK 기반 관계 자동 생성(entities.js)은 양방향을 모두 중복으로 처리한다. 두 경로의 중복 판단 기준을 통일하면 혼동을 줄인다. | js/relations.js |
| low | dangling 참조 관계 편집 시 from/to 무음 재할당 위험 — openEditRelationModal→populateEntitySelects에서 rel.from/rel.to가 현재 ENTITIES에 없으면 셀렉트에 일치 옵션이 없어 첫 엔티티가 기본 선택되고, 그대로 저장 시 관계가 조용히 다른 엔티티로 바뀐다. 일치 옵션이 없을 때 경고 표시 또는 누락 엔티티용 placeholder 옵션 추가를 권장한다. | js/relations.js |
| low | 관계 색상 편집이 모달에만 있고 인라인 속성 패널엔 없음 — relOverlay 모달에서는 선 색상(relColor)을 지정할 수 있으나 ui.js의 인라인 관계 속성 패널(_renderRelPropPanel)에는 색상 편집 UI가 없다. 일관성을 위해 인라인 패널에도 색상 편집을 추가하면 UX가 개선된다. | js/ui.js |
| low | FK 자동 추가 시 PK 부재 엔티티의 ref.attr 기본값 'ID' 가정 — autoAddFkColumn은 시작 엔티티에 PK가 없으면 ref.attr을 'ID'로 가정한다. 실제로 ID 컬럼이 없는 스키마에서는 존재하지 않는 컬럼을 참조하는 FK가 만들어진다. PK가 없을 때 FK 자동 추가를 건너뛰거나 사용자에게 PK 지정을 안내하는 편이 견고하다. | js/entities.js |

## 4. 추천 사항

- **관계 모달에 키보드 단축키(Enter 저장 / Esc 취소) 지원**: 현재 relOverlay 모달은 저장/취소를 버튼 클릭으로만 처리한다. Enter→saveRelation, Esc→closeRelModal 키 핸들러를 추가하면 다른 입력 모달과의 일관성과 입력 효율이 향상된다. 입력 필드 포커스 상태에서 'R' 단축키가 동작하지 않도록 처리한 기존 정책과도 잘 맞는다.
- **관계 무결성 검사/정리 유틸 제공**: 가져오기·공유·병합 후 ENTITIES에 존재하지 않는 from/to를 가진 dangling 관계를 검출·정리하는 진단/자동 정리 기능을 추가하면 무음 재할당을 근본적으로 예방할 수 있다. 정규화 진단 패널과 함께 노출하면 좋다.
- **FK 자동 추가 결과 토스트 알림**: FK 컬럼 자동 추가가 실제로 컬럼을 삽입했는지(이미 존재해 건너뛴 경우 포함) showToast로 피드백하면, 사용자가 끝 엔티티에 컬럼이 생겼는지 즉시 확인할 수 있어 화면 갱신 누락 같은 혼동을 줄인다.
