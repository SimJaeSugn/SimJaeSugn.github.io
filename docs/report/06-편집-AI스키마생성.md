# 편집 ▸ AI 스키마 생성 — 분석 보고서

## 1. 개요

"편집 ▸ AI 스키마 생성" 메뉴의 실제 구현은 `js/ui.js`가 아니라 `js/import.js`(397~501행)에 있으며, `openAISchemaModal` / `closeAISchemaModal` / `runAISchemaGen` / `applyAISchema` 네 함수로 구성된다. `index.html`의 메뉴·모달·`overlayCloseExtra`(`ui.js`) 연결, `AI_KEY_STORAGE`(`config.js`) 로컬 저장, 전체 백업(`export.js`)·복원(`import.js`) 통합은 모두 정상이며, 단축키 충돌이나 `onclick` 오타도 없다. Anthropic API 호출(`fetch`), 로딩/에러 처리, JSON 파싱, 적용 방식(추가/교체), `saveState()` / `render()` / `renderEntityTree()` 호출도 대체로 견고하다. 다만 엔티티 id 중복 시 관계의 `from`/`to`를 함께 갱신하지 않아 관계가 끊기는 데이터 무결성 버그가 발견되어 수정 완료하였으며, 교체 모드에서의 선택 상태 잔존 문제와 AI 생성 객체의 표준 필드 누락 일관성 문제도 검토되었다.

## 2. 발견된 오류 및 수정 결과

### [medium] 엔티티 id 중복 재명명 시 관계 from/to 미갱신 → 관계 유실(끊김)

- **상태**: 수정완료
- **증상**: `applyAISchema`는 먼저 `relations` 배열을 AI가 준 원본 엔티티 id(`r.from`/`r.to`)로 만든 뒤, 엔티티를 추가하면서 id 충돌이 나면 `e.id`를 `'foo_2'`처럼 재명명한다. 그러나 `relations`의 `from`/`to`는 원본 id 그대로 남아, 재명명된 엔티티를 가리키던 관계는 존재하지 않는 id를 참조하게 된다. 결과적으로 그 관계는 `getRelationPath`가 `null`을 반환해 캔버스에 그려지지 않고(보이지 않게 사라짐), `RELATIONS` 배열·전체 백업·DDL 내보내기에는 깨진 참조로 남는다. 재현: (1) '현재 다이어그램에 추가' 모드에서 기존 다이어그램에 `customer` 엔티티가 이미 있는 상태로 AI가 다시 `customer`를 포함한 스키마를 생성하면, 새 `customer`는 `customer_2`로 바뀌지만 관계는 여전히 `customer`를 가리켜 끊긴다. (2) AI 응답 자체에 동일 id 엔티티가 둘 이상 있어도 같은 문제가 발생한다. 즉 사용자가 본 '관계 N개'와 실제 연결된 관계 수가 달라지고 데이터가 조용히 손상된다.
- **원인**: 엔티티 dedup 루프에서 `e.id`를 재명명하지만 `oldId → newId` 매핑을 만들어 `relations`의 `from`/`to`에 적용하지 않음. 관계 push는 변환되지 않은 원본 id를 사용.
- **수정 내용**: AI 스키마 적용 시 `idRemap`으로 엔티티 ID 재명명을 추적하여 `relations`의 `from`/`to`를 올바른 ID로 갱신하고, 유효하지 않은 관계를 필터링하도록 수정.
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\import.js`

### [low] 교체(replace) 모드에서 선택 상태(selectedEntity/selectedRelation) 미초기화로 삭제된 엔티티 참조 잔존

- **상태**: 오류아님
- **증상**: '현재 다이어그램 교체' 선택 시 `ENTITIES.length=0; RELATIONS.length=0`으로 작업 배열만 비우고 `selectedEntity`/`selectedRelation`은 그대로 둔다. 직전에 엔티티/관계를 선택해 둔 상태에서 교체를 실행하면 이들 전역 변수는 더 이상 `ENTITIES`/`RELATIONS`에 존재하지 않는 객체를 가리키는 댕글링 참조가 된다. 캔버스 하이라이트·우측 속성 패널이 삭제된 엔티티 정보를 계속 표시하거나, 그 상태에서 편집/삭제 같은 후속 동작이 이미 없어진 객체를 대상으로 하여 예기치 않게 동작할 수 있다.
- **원인**: `replace` 분기에서 작업 배열만 비우고 `selection` 전역(`selectedEntity`, `selectedRelation` 등)을 `null`로 리셋하지 않음.
- **수정 내용**: 주장의 핵심 전제가 사실과 다르므로 오류아님으로 판정. 비교 기준인 "정상 경로" `loadDiagramIntoWorkspace`(`js/state.js:70-80`)는 `selectedSection`/`selectedSections`만 `null`로 초기화하며 `selectedEntity`/`selectedRelation`/`selectedEntities`는 초기화하지 않는다. 즉 다이어그램 전환 경로도 `replace`와 똑같이 stale 참조를 남기므로 "이 경로만 일관성이 깨진다"는 주장은 사실이 아니다. 또한 stale 참조가 실제 깨짐·유실을 일으키지 않는다(`drawEntity`는 동일성 비교라 dangling 객체와 절대 일치하지 않고, `deleteEntity`는 `ENTITIES.indexOf(entity)===-1`이라 `splice`가 no-op으로 안전하며, `saveState`는 `ENTITIES`만 직렬화하므로 dangling 객체는 영속화되지 않음). 속성 패널은 `selectedEntity`/`selectedRelation`이 아니라 별도 전역 `_ppMode`/`_ppTarget`(`js/ui.js:1138-1139`)으로 동작하므로 제안 수정이 주장한 증상도 해결하지 못한다. 일관성을 위해 정말 손대려면 `import.js`만이 아니라 `loadDiagramIntoWorkspace`에서 `selectedEntity`/`selectedRelation`/`selectedEntities`/`_ppTarget`까지 함께 초기화하는 더 넓은 변경이 필요하므로 국소 자동 수정 대상이 아니다.
- **파일**: `js/import.js`

## 4. 추천 사항

- **AbortController로 요청 취소 지원**: 네트워크 지연 시 사용자가 취소할 수 있도록 `fetch`에 `AbortController.signal`을 연결하고, '취소' 버튼이나 모달 닫기 시 `abort()`를 호출해 응답이 와도 적용되지 않게 한다.

- **적용을 saveState 단일 트랜잭션으로 처리하고 undo 1회 보장**: `applyAISchema`는 끝에서 `saveState()`를 한 번 호출하므로 현재도 undo 1회로 되돌아가지만, 다수 엔티티/관계 추가가 의도대로 한 스텝으로 묶이는지 회귀 테스트를 추가하면 협업/타임라인과의 충돌을 조기에 발견할 수 있다.

- **모델명/토큰 한도 설정값 외부화**: `model:'claude-haiku-4-5-20251001'`, `max_tokens:2048`이 하드코딩되어 있다. 큰 스키마는 2048 토큰으로 잘려 JSON 파싱이 실패할 수 있으므로, 모델/토큰을 `config.js` 상수로 빼고 필요 시 `max_tokens`를 늘리거나 모달에서 선택 가능하게 한다.

- **브라우저 직접 호출(CORS/키 노출) 한계 안내 강화**: `anthropic-dangerous-direct-browser-access`로 브라우저에서 직접 호출하므로 API 키가 로컬스토리지에 평문 저장되고 네트워크 탭에 노출된다. 모달 안내 문구에 보안 주의(개인용/신뢰 환경 한정)와, 가능하면 middleware 프록시 경유 옵션을 권장한다.

- **AI 응답 스키마 유효성 검증 단계 추가**: 적용 전에 `entities[].id` 중복, 관계의 `from`/`to`가 실제 엔티티를 가리키는지, pk 최소 1개 보유 등을 검증해 사용자에게 미리보기/경고를 보여주면(예: DDL 가져오기 미리보기처럼) 데이터 무결성과 신뢰도가 높아진다.
