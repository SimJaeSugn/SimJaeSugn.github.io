# 보기 ▸ 테마·표기법·뷰 — 분석 보고서

## 1. 개요

"보기 ▸ 테마·표기법·뷰" 메뉴는 대부분 정상 동작한다. 테마(applyTheme/openThemeModal/loadSavedTheme)는 THEME_STORAGE 영속화, 전체백업 settings.theme, 구 테마명 마이그레이션까지 잘 갖춰져 있으며, 모달 닫기(classList.remove('active'))도 .modal-overlay.active CSS 규약과 일치한다. 줌(zoom/updateZoomLabel)은 두 곳의 줌 표시(zoomLabel, sb-zoom-stat)가 render()→updateStatusBar() 경유로 동기화되며, resetView/fitAll도 동작한다. 다만 표기법 토글(toggleNotation)과 논리/물리 전환(setViewMode)이 영속/언두 직렬화 대상(notationStyle·viewMode)을 변경하면서도 saveState()를 호출하지 않아 새로고침·실행취소 시 변경이 유실되는 영속성 무결성 위반 2건이 확인됐으며, 단축키(fitAll=Home)는 main.js·shortcuts.js·모달 표시가 일치한다.

## 2. 발견된 오류 및 수정 결과

### [low] 크로우풋 표기 토글이 saveState()를 호출하지 않아 새로고침·실행취소 시 유실

- **상태**: 오류아님
- **증상**: 메뉴 '보기 ▸ 크로우풋 표기' 또는 커맨드 팔레트 '크로우풋 표기'를 눌러 notationStyle을 바꾸면 화면은 즉시 다시 그려지지만(render), 변경된 notationStyle이 localStorage(STORAGE_KEY)에 저장되지 않는다. 따라서 표기법을 크로우풋으로 바꾼 뒤 페이지를 새로고침하면 simple로 되돌아간다. 또한 undo 스택에도 push되지 않아 이 토글은 실행취소가 불가능하고, 이후 다른 작업으로 saveState()가 호출될 때 비로소 바뀐 notationStyle이 조용히 스냅샷에 섞여 들어가 직전 언두 상태들과 불일치가 생긴다. notationStyle은 saveState()의 직렬화 대상({ diagrams, activeDiagramId, viewMode, notationStyle, gridSnap })에 포함되어 있으므로 영속/언두가 의도된 상태인데, 변경 시점에 저장 트리거가 빠진 것이다. (영속성 무결성 규칙 위반: 저장 메커니즘은 존재하나 변이 지점에서 호출 누락.)
- **원인**: toggleNotation()이 notationStyle 변경 후 syncToolDropdownLabels()와 render()만 호출하고 saveState()(=flushCurrentState + undo push + localStorage 저장)를 호출하지 않는다. toggleNotation은 사용자 액션(메뉴/커맨드 팔레트)에서만 호출되며 restoreFromSnapshot/초기화 경로에서는 호출되지 않으므로 함수 내부에 saveState()를 추가해도 안전하다.
- **수정 내용**: anchor 검증 결과 js/ui.js 666~670행에 해당 anchor가 원문 그대로·유일하게 존재함을 확인했다. notationStyle은 saveState() 직렬화 대상(state.js:84)이고 localStorage(STORAGE_KEY)에 저장(state.js:94)되며, undo/redo restore(state.js:130)와 init loadState(state.js:164)에서 복원된다. toggleNotation()은 saveState()를 호출하지 않으므로 표기 토글만 단독으로 한 뒤 새로고침하면(이후 다른 저장 트리거가 없을 경우) 되돌아간다. 그러나 적대적 검증 결과 '비정상적 누락/무결성 규칙 위반'이라는 판단은 성립하기 어렵다. setViewMode()(state.js:215), toggleGridSnap()(ui.js:659~663), toggleNotation()(ui.js:666~670) 세 함수 모두 동일하게 syncToolDropdownLabels()/render()만 호출하고 saveState()를 호출하지 않는 일관된 패턴이며, 이들은 다음 실제 데이터 편집 시 스냅샷에 함께 실려가는 '전이적 뷰 옵션'으로 설계된 것으로 보인다. toggleNotation만 잘못 누락된 것이 아니라 형제 토글 2개와 완전히 동일하므로 결함이라기보다 의도된 설계로 판단된다. 자동 수정 미적용.
- **파일**: js/ui.js

### [medium] 논리/물리 보기 전환이 saveState()를 호출하지 않아 새로고침·실행취소 시 유실

- **상태**: 수정실패
- **증상**: 메뉴 '보기 ▸ 논리 보기 / 물리 보기'(index.html 80~81행), 퀵바 '논리/물리' 버튼(index.html 169~170행, onclick="setViewMode('logical'|'physical')"), 커맨드 팔레트의 '논리 보기'/'물리 보기'(js/ui.js 2274~2275행 action: () => setViewMode(...))를 눌러 viewMode를 바꾸면 화면(render/renderEntityTree)과 메뉴 체크표시(syncToolDropdownLabels)는 갱신되지만, 변경된 viewMode가 localStorage에 저장되지 않고 undo 스택에도 push되지 않는다. 그 결과 논리↔물리 전환 후 새로고침하면 항상 logical(또는 마지막으로 saveState된 값)로 되돌아가고, 전환 자체는 실행취소가 불가능하다. viewMode는 saveState()/loadState()/restoreFromSnapshot()의 직렬화 대상에 포함되어 있어 영속·언두가 의도된 상태인데, 사용자 전환 시점에 저장 트리거가 빠져 있다.
- **원인**: viewMode를 변경하는 사용자 진입점(메뉴/퀵바/팔레트 onclick)이 setViewMode(mode)만 호출하고 saveState()를 호출하지 않는다. setViewMode 자체는 복원/초기화/원격동기화 공용 경로라 saveState를 넣을 수 없으므로, 영속화 트리거는 각 사용자 액션 호출 지점에 추가되어야 한다.
- **수정 내용**: 이미 적용됨 — 파일 80~81번 줄에 saveState() 호출이 이미 존재하여 변경이 필요 없음. 다만 퀵바 버튼(index.html 169~170행)과 커맨드 팔레트 action(js/ui.js 2274~2275행)에 대한 보강은 미완료 상태로 남아 있다.
- **파일**: E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\index.html

## 4. 추천 사항

- **영속 상태 변이 지점에 saveState 호출을 강제하는 단일 진입점 패턴**: viewMode·notationStyle·gridSnap처럼 saveState 직렬화 대상인 전역 상태는 토글 시 saveState 누락이 반복적으로 발생하고 있다(e1, e2, toggleGridSnap). 각 상태마다 '순수 setter(복원/동기화용)'와 '사용자 액션용 setter(setter + saveState)'를 분리하거나, 토글 함수가 항상 saveState로 끝나도록 컨벤션을 코드 리뷰 체크리스트(CLAUDE.md 하네스)에 명시하면 동일 클래스 버그를 구조적으로 방지할 수 있다.

- **표기법/보기 모드 토글에 대한 회귀 테스트(새로고침 영속성) 추가**: 번들러가 없는 순수 JS 프로젝트라도, '상태 변경 → STORAGE_KEY 직렬화 문자열에 반영되는가'를 검증하는 가벼운 수동/스크립트 점검 절차를 두면 saveState 누락을 조기에 잡을 수 있다. 최소한 toggleNotation/setViewMode 호출 후 localStorage[STORAGE_KEY]에 변경값이 들어있는지 확인하는 항목을 추가 권장한다.
