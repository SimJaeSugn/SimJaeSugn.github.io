# 도구 ▸ 스냅샷·타임라인 — 분석 보고서

## 1. 개요

스냅샷·타임라인 메뉴는 js/ui.js(스냅샷 저장/목록/복원/삭제, 최대 20개), js/diagrams.js(openDiffModal diff 비교), js/timeline.js(하단 HUD 슬라이더·미리보기·복원·닫기 시 원본복구)로 구성된다. SNAPSHOTS 배열은 별도 localStorage 키(erd_snapshots)로 영속화되며 전체 백업(export.js/import.js)에도 포함되어 영속성 무결성은 충족한다. 핵심 함수 연결(onclick→전역함수)과 escHtml 이스케이프, 미리보기 시 JSON.parse try/catch 처리는 대체로 견고하나, 미리보기 모드에서 전역 keydown 단축키가 차단되지 않아 미리보기 데이터가 영속화되는 문제가 핵심 결함으로 확인되었다. 전반적으로 동작은 하나 미리보기-전역단축키 격리가 미흡하여 README가 보장한 '복원 전 데이터 불변' 원칙이 깨질 수 있다.

## 2. 발견된 오류 및 수정 결과

### [High] 타임라인 미리보기 중 전역 단축키가 차단되지 않아 미리보기 데이터가 영속화됨 (README 보장 위반)

- **상태**: 수정완료
- **증상**: 타임라인 HUD를 열면 _tlPreviewMode=true 상태로 ENTITIES/RELATIONS가 선택한 스냅샷 데이터로 교체된다(미리보기). 하지만 main.js의 전역 keydown 핸들러는 _tlPreviewMode를 확인하지 않으므로 미리보기 중에도 화살표(엔티티 이동), Delete(삭제), Ctrl+D(복제), Ctrl+A 등이 그대로 동작한다. 슬라이더에 포커스가 없으면(openTimelineModal이 슬라이더를 focus하지 않으므로 포커스는 body/canvas에 남음) 입력필드 가드(line 58)를 통과해 단축키가 실행된다. 특히 화살표 키는 main.js 핸들러가 먼저 등록되어 먼저 실행되므로, 미리보기 중 선택된 엔티티가 있으면 한 번의 화살표 입력으로 (a) 미리보기 엔티티를 이동시키고 (b) render()+saveState()로 미리보기/스냅샷 데이터를 활성 다이어그램에 flush 후 localStorage에 영속화한다. 사용자가 '↩ 복원'을 누르지 않았는데도 현재 작업이 스냅샷 데이터로 덮어써져 데이터 유실이 발생한다. README 8장은 '복원 버튼을 누르지 않는 한 실제 데이터는 변경되지 않습니다'를 명시하므로 명백한 사양 위반이자 데이터 무결성 버그다. 또한 같은 화살표 입력이 HUD 슬라이더 이동과 동시에 처리되는 단축키 충돌도 함께 발생한다.
- **원인**: 전역 keydown 핸들러가 타임라인 미리보기 모드(_tlPreviewMode)를 인지하지 못해, 미리보기 중 ENTITIES/RELATIONS가 임시(비영속) 상태인데도 saveState()를 호출하는 단축키들을 그대로 허용한다. 미리보기 상태와 실제 작업 상태의 격리가 단축키 레이어에 없다.
- **수정 내용**: keydown 핸들러에서 INPUT/TEXTAREA/SELECT 차단 직후 _tlPreviewMode 가드를 추가하여 타임라인 미리보기 중 전역 단축키를 차단함
- **파일**: E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\main.js

### [Low] _hideTimelineHud의 setTimeout 경쟁으로 재오픈된 HUD가 숨겨질 수 있음

- **상태**: 수정완료
- **증상**: closeTimeline()/restoreTimelineSnapshot()는 _hideTimelineHud()를 호출하고, 이 함수는 200ms 후 hud.style.display='none'을 설정하는 setTimeout을 건다. 이 타이머는 어디에도 저장/취소되지 않는다. 닫은 직후 200ms 이내에 다시 openTimelineModal()을 호출하면(메뉴 더블클릭, 닫고 바로 재오픈 등) openTimelineModal은 display=''로 HUD를 표시하고 _tlPreviewMode=true로 만들지만, 이전 닫기에서 예약된 setTimeout이 뒤늦게 발화하여 방금 연 HUD를 display:none으로 숨긴다. 결과적으로 HUD가 보이지 않는데 _tlPreviewMode=true·ENTITIES는 미리보기 상태로 남아 사용자가 닫을 수단(✕/Esc)이 화면에 없는 깨진 상태가 된다.
- **원인**: 숨김 애니메이션 완료용 setTimeout 핸들을 보관/취소하지 않아, 재오픈 시 이전 닫기 타이머가 살아남아 새 HUD에 display:none을 적용하는 타이머 경쟁이 발생한다.
- **수정 내용**: _hideTimelineHud에 _tlHideTimer 변수를 도입해 clearTimeout으로 기존 타이머를 취소하고, 콜백 실행 시 _tlPreviewMode가 true이면 display:none을 건너뜀으로써 재오픈된 HUD가 setTimeout 경쟁으로 숨겨지는 버그를 수정했다.
- **파일**: E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\timeline.js

### [Low] restoreTimelineSnapshot의 JSON.parse/복원이 try/catch 없이 실행되어 손상 스냅샷에서 조용히 실패

- **상태**: 수정완료
- **증상**: restoreTimelineSnapshot()는 askConfirm 콜백에서 restoreFromSnapshot(JSON.parse(snap.state))를 try/catch 없이 호출한다. snap.state가 손상된 JSON이면 JSON.parse가 예외를 던지고 이후 saveState()·_hideTimelineHud()·render()가 실행되지 않아, 복원이 조용히 실패하고 HUD가 닫히지 않은 채 _tlPreviewMode가 true로 남는다(미리보기로 ENTITIES는 이미 교체된 상태). 동일 기능을 하는 js/ui.js의 restoreSnapshot()은 try/catch로 감싸고 실패 시 토스트를 띄우므로 동작이 일관되지 않다.
- **원인**: 형제 함수(ui.js restoreSnapshot)와 달리 JSON.parse 및 복원 호출을 예외 처리로 보호하지 않아, 손상 데이터 입력 시 복원 파이프라인(상태 저장·HUD 닫기·렌더)이 중단된다.
- **수정 내용**: restoreTimelineSnapshot의 JSON.parse 및 복원 로직 전체를 try/catch로 감싸 손상된 스냅샷에서 오류 발생 시 사용자에게 토스트 메시지로 알리도록 수정
- **파일**: js/timeline.js

## 3. 개선점 (자동수정 안 함 — 권장)

| 우선순위 | 제목 | 파일 |
|---------|------|------|
| Medium | 미리보기 중 Enter 재입력 시 복원 확인창이 중복 호출됨 | js/timeline.js |
| Low | Esc로 HUD를 닫을 때 전역 Escape 핸들러가 함께 실행되어 선택이 초기화됨 | js/timeline.js |
| Low | 미리보기 시 좌측 엔티티 트리가 갱신되지 않아 캔버스와 불일치 | js/timeline.js |
| Low | 미리보기는 ENTITIES/RELATIONS만 교체하여 SECTIONS/NOTES가 현재본으로 남음 | js/timeline.js |
| Low | openTimelineModal이 슬라이더에 포커스를 주지 않아 키 입력 흐름이 모호함 | js/timeline.js |

- **미리보기 중 Enter 재입력 시 복원 확인창이 중복 호출됨**: HUD _keyHandler에서 Enter는 _tlPreviewMode가 true인 한 restoreTimelineSnapshot()를 호출한다. 확인(askConfirm) 모달이 열려 있어도 _tlPreviewMode는 여전히 true이므로 Enter를 누를 때마다 askConfirm이 다시 호출되어 pendingConfirmFn이 덮어써지고 모달이 재표시된다. 또한 확인창이 떠 있는 동안 화살표 키도 여전히 동작하여 _tlSelectedIdx가 바뀌면, 확인창이 안내하는 스냅샷 이름과 실제 복원 대상이 달라질 수 있다. 확인창이 열려 있는 동안에는 HUD 키 핸들러가 동작하지 않도록 .modal-overlay.active 체크를 추가하는 것이 안전하다.
- **Esc로 HUD를 닫을 때 전역 Escape 핸들러가 함께 실행되어 선택이 초기화됨**: HUD _keyHandler의 Escape 처리는 closeTimeline() 호출 후 e.preventDefault()/stopPropagation()을 하지 않아 main.js의 전역 Escape 핸들러도 실행된다. 그 결과 selectedEntities.clear()와 추가 render()가 일어나, ✕ 버튼(onclick=closeTimeline)으로 닫을 때와 동작이 달라진다(Esc로 닫으면 선택 해제, 버튼으로 닫으면 선택 유지). 일관성을 위해 Esc 처리 시 closeTimeline() 후 이벤트 전파를 멈추는 것이 좋다.
- **미리보기 시 좌측 엔티티 트리가 갱신되지 않아 캔버스와 불일치**: onTimelineSlide()는 캔버스만 renderNow()로 갱신하고 renderEntityTree()는 호출하지 않는다. 미리보기로 캔버스에는 스냅샷 시점 엔티티가 표시되지만 좌측 트리/상태바는 현재 작업본 기준으로 남아 화면-상태가 불일치한다. 미리보기 시 트리/상태바도 임시 갱신하거나, 의도된 동작이면 README에 명시하는 것이 좋다.
- **미리보기는 ENTITIES/RELATIONS만 교체하여 SECTIONS/NOTES가 현재본으로 남음**: onTimelineSlide()는 ENTITIES/RELATIONS만 스냅샷 값으로 바꾸고 SECTIONS/NOTES/NOTES_V2는 현재 작업본을 그대로 둔다. 따라서 미리보기 캔버스에 스냅샷의 엔티티 위에 현재 섹션/메모가 겹쳐 보일 수 있다. 미리보기 정확도를 높이려면 섹션/메모도 함께 임시 반영하고 closeTimeline에서 원복하는 것을 고려할 수 있다.
- **openTimelineModal이 슬라이더에 포커스를 주지 않아 키 입력 흐름이 모호함**: HUD를 열어도 슬라이더에 포커스가 가지 않으므로 화살표 키 처리는 document 레벨 핸들러에 의존한다. HUD 열 때 슬라이더에 focus()를 주면 의도가 분명해지고, main.js의 INPUT 포커스 가드 덕분에 전역 단축키와의 충돌도 자연스럽게 줄어든다.

## 4. 추천 사항

- **미리보기 상태 격리를 단일 플래그/함수로 캡슐화**: _tlPreviewMode 진입/해제를 enterPreview()/exitPreview() 같은 헬퍼로 묶고, 전역 keydown·메뉴 동작·자동저장(broadcast/webrtc 협업 진입 시 스냅샷 등)이 모두 이 플래그를 존중하도록 하면 미리보기 중 의도치 않은 영속화를 구조적으로 방지할 수 있다.
- **스냅샷 복원 시 자동 백업 스냅샷 생성**: restoreTimelineSnapshot/restoreSnapshot에서 복원 직전 현재 상태를 '복원 전 자동저장' 스냅샷으로 unshift하면(협업 진입 시 webrtc.js가 하는 방식과 동일) Ctrl+Z 외에 명시적 되돌림 지점을 제공해 안전성이 높아진다.
- **diff 비교에 관계 카디널리티/이름 변경도 반영**: openDiffModal의 관계 비교는 from/to만 보고 card 변경을 감지하지 못한다. card나 관계 라벨 변경도 '변경된 관계'로 표시하면 diff 정확도가 향상된다.
- **타임라인 HUD 키 동작을 단축키 모달(shortcutsOverlay)에 컨텍스트 항목으로 노출**: 현재 ←/→/Enter/Esc는 HUD 힌트와 README에만 있다. 단축키 도움말 모달에 'HUD 활성 시' 컨텍스트 섹션으로 함께 보여주면 발견성이 좋아진다.
