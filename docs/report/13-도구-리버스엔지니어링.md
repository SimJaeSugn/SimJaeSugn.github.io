# 도구 ▸ 리버스 엔지니어링 — 분석 보고서

## 1. 개요

"도구 ▸ 리버스 엔지니어링" 메뉴는 미들웨어(/schema/tables, /schema)를 통해 실제 DB 스키마를 읽어 ERD를 생성한다. 2단계 모달(옵션 → 테이블 선택)로 동작하며, 새 다이어그램 생성·현재 덮어쓰기·현재에 추가 3가지 모드와 대문자 변환 옵션을 지원한다. 전반적으로 구조가 잘 잡혀 있고 saveState/flush/loadDiagramIntoWorkspace 호출 순서도 'new' 모드에서는 정확하다. 다만 덮어쓰기 모드에서 기존 엔티티의 physicalName이 없을 때 null 역참조로 생성 전체가 중단되는 문제와, 테이블 목록 체크리스트에서 DB 식별자를 이스케이프 없이 삽입하여 특수문자 포함 식별자가 깨지거나 선택에서 누락되는 문제가 실제 버그로 확인되어 수정되었다.

## 2. 발견된 오류 및 수정 결과

### [medium] 덮어쓰기 모드: 기존 엔티티 physicalName 미존재 시 null 역참조로 ERD 생성 전체 중단

- **상태**: 수정완료
- **증상**: 리버스 엔지니어링의 '현재 다이어그램 덮어쓰기' 모드에서 기존 엔티티 위치를 보존하려고 d.entities를 순회하며 e.physicalName.toLowerCase()를 호출한다. 그런데 외부 JSON import 경로(js/import.js 132줄: d.entities = data.entities)나 구버전 백업으로 들어온 엔티티는 physicalName 필드가 없거나 null일 수 있다. 이 경우 toLowerCase() 호출에서 TypeError가 발생하고, 이 코드는 try 블록 안이라 catch로 떨어져 'ERD 생성 완료'가 아닌 오류 메시지가 표시되며 스키마 읽기·엔티티 생성이 통째로 무산된다. 즉 현재 다이어그램에 physicalName 없는 엔티티가 하나라도 있으면 덮어쓰기 모드가 항상 실패한다.
- **원인**: posMap 구성 시 e.physicalName이 항상 문자열이라고 가정. optional/누락 필드에 대한 방어가 없음(null/undefined 역참조).
- **수정 내용**: 덮어쓰기 모드 posMap 생성 및 조회 시 physicalName이 null/undefined일 때 logicalName으로 폴백하여 null 역참조 오류 수정
- **파일**: js/reverse_engineer.js

### [medium] 테이블 목록 체크리스트: DB 식별자 비이스케이프 삽입으로 렌더 깨짐·선택 누락

- **상태**: 수정완료
- **증상**: 1단계에서 받은 테이블/뷰 이름(it.name)을 escHtml 없이 innerHTML의 data-name 속성과 &lt;span&gt; 안에 그대로 삽입한다. 식별자에 큰따옴표(")가 포함되면(예: MySQL 백틱/MSSQL 대괄호 식별자 my"tbl) data-name="my"tbl"가 깨져 getAttribute('data-name')가 잘린 값(my)을 반환한다. 2단계에서 selectedSet.has(t.tableName)으로 필터링할 때 원본 이름(my"tbl)과 매칭되지 않아 해당 테이블이 조용히 ERD에서 누락된다. &lt;가 포함되면 체크박스 라벨 자체가 깨진다. 코드베이스의 다른 부분(entities.js)은 사용자/외부 문자열을 innerHTML에 넣을 때 escHtml을 일관되게 사용한다.
- **원인**: DB에서 받은 식별자를 신뢰 가능한 문자열로 간주하여 escHtml 없이 innerHTML/속성에 직접 보간. data-name 속성 깨짐이 2단계 선택 집합과의 불일치로 이어짐.
- **수정 내용**: 테이블 목록 체크리스트 렌더링 시 it.name과 it.type에 escHtml 적용하고 data-name 속성에 &amp;quot; 이스케이프 추가하여 XSS 및 렌더 깨짐 수정
- **파일**: js/reverse_engineer.js

## 3. 개선점 (자동수정 안 함 — 권장)

| 우선순위 | 제목 | 파일 |
|----------|------|------|
| medium | append/overwrite 분기에서 flushCurrentState() 선반영 누락 | js/reverse_engineer.js |
| low | 추가/덮어쓰기 모드에서 getActiveDiagram() undefined 방어 부재 | js/reverse_engineer.js |
| low | 모달 재오픈 시 라디오/이름행 상태 초기화 누락 | js/reverse_engineer.js |
| low | VIEW 메모 격자 X 오프셋 상수 불일치로 엔티티와 겹칠 수 있음 | js/reverse_engineer.js |
| low | append 모드 관계 중복 제거가 단방향만 검사 | js/reverse_engineer.js |
| low | PostgreSQL 스키마 문자열 치환 SQL 인젝션 표면(미들웨어) | middleware/src/routes/schema.js |

**상세 내용:**

- **append/overwrite 분기에서 flushCurrentState() 선반영 누락 (medium)**: 'new' 모드는 diagrams에 push하기 전에 flushCurrentState()로 현재 작업 배열을 활성 다이어그램에 반영한다. 반면 'append'/'overwrite'는 getActiveDiagram()으로 d를 읽고 d.entities/d.notesV2를 직접 사용하는데, 직전에 flush를 호출하지 않는다. 마지막 저장 이후 발생한 미저장 상태(예: 패닝 vx/vy)는 loadDiagramIntoWorkspace(d) 시 되돌아갈 수 있다. 분기 진입부에서 flushCurrentState()를 호출하면 일관성이 보장된다.

- **추가/덮어쓰기 모드에서 getActiveDiagram() undefined 방어 부재 (low)**: 'append'/'overwrite' 분기는 const d = getActiveDiagram() 결과를 곧바로 d.entities 등으로 역참조한다. getActiveDiagram()은 diagrams가 비면 undefined를 반환할 수 있다. 앱이 항상 1개 이상의 다이어그램을 유지하므로 현재는 도달 불가에 가깝지만, 방어 코드(if (!d) { ...적절한 안내 후 return; })를 두면 견고성이 향상된다.

- **모달 재오픈 시 라디오/이름행 상태 초기화 누락 (low)**: _renderReverseEngineerModal()은 오버레이가 이미 있으면 early-return하여 DOM(라디오 선택, reNewNameRow 표시 여부)이 이전 세션 상태로 유지된다. _resetReverseEngineerUI()는 옵션/목록 표시만 되돌릴 뿐 reMode 라디오를 'new'로, reNewNameRow를 표시 상태로 초기화하지 않는다. 이전에 '덮어쓰기'를 골랐다면 재오픈 시에도 그 선택과 숨겨진 이름행이 남아 사용자 혼란을 줄 수 있다.

- **VIEW 메모 격자 X 오프셋 상수 불일치로 엔티티와 겹칠 수 있음 (low)**: _buildViewNotes의 OFFSET_X = 60 + 5 * 220 + 60은 COL_W를 220으로 가정하지만 _buildEntitiesFromSchema의 실제 COL_W는 240, GAP_X는 30이다. 따라서 엔티티 격자 우측 끝(60 + 5*(240+30) = 1410)보다 VIEW 메모 시작 X(1210)가 더 왼쪽이라 뷰가 많을 때 메모가 마지막 엔티티 열과 겹칠 수 있다. 동일 상수(COL_W/GAP_X/COLS)를 공유하도록 정리를 권장한다.

- **append 모드 관계 중복 제거가 단방향만 검사 (low)**: merged.find(x => x.from === r.from && x.to === r.to)는 from/to가 뒤바뀐 동일 관계(역방향)를 중복으로 보지 않는다. 자기 자신 다이어그램에 같은 스키마를 반복 append하면 방향이 다른 동일 FK가 중복 추가될 수 있다.

- **PostgreSQL 스키마 문자열 치환 SQL 인젝션 표면(미들웨어) (low)**: schema.js의 PG_* 쿼리는 파라미터 바인딩 대신 화이트리스트(^[A-Za-z0-9_.]+$) 통과 후 스키마명을 문자열로 직접 치환한다. 정규식이 인용부호·세미콜론을 차단하므로 현재는 안전하나, 점(.)을 허용해 schema.table 형태가 통과될 수 있어 잠재적 표면이 남는다. 가능하면 어댑터 파라미터 바인딩으로 전환하거나 점 허용 범위를 좁히는 것을 권장한다.

## 4. 추천 사항

- **테이블 검색/필터 입력란 추가**: 테이블이 수십~수백 개인 DB에서는 2단계 체크리스트에서 원하는 테이블을 찾기 어렵다. 목록 상단에 검색 입력(이름 부분일치 필터)과 '뷰만/테이블만' 토글을 추가하면 대규모 스키마에서 사용성이 크게 향상된다.

- **컬럼 길이/스케일 등 dataType 정밀도 반영**: 미들웨어는 character_maximum_length(PG/MSSQL/Oracle) 등을 조회하지만 클라이언트 _buildEntitiesFromSchema는 c.dataType만 사용한다. VARCHAR(n), NUMERIC(p,s)처럼 길이/정밀도를 type 문자열에 합성하면 생성된 ERD의 컬럼 타입이 실제 스키마와 더 정확히 일치한다.

- **생성 후 자동 정렬/뷰 맞춤(fit) 호출**: ERD 생성 직후 격자 배치만 적용된다. 생성 완료 후 전체 보기에 맞추는 fitToScreen류 동작이나 기존 자동 레이아웃을 옵션으로 호출하면, 특히 'new' 모드에서 사용자가 즉시 전체 구조를 파악하기 쉽다.

- **복합 FK(다중 컬럼) 관계 카디널리티 정합성**: FK 카디널리티(1:1/1:N)를 from 컬럼의 UNIQUE 여부 단일 컬럼 기준으로 추정한다. 복합 키 FK의 경우 컬럼 단위 uniqueSet 매칭이 부정확할 수 있으므로, 제약명 단위로 컬럼 집합을 묶어 카디널리티를 판정하면 정확도가 올라간다.
