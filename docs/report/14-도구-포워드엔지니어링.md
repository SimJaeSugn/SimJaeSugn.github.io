# 도구 ▸ 포워드 엔지니어링 — 분석 보고서

## 1. 개요

"도구 ▸ 포워드 엔지니어링" 메뉴(js/forward_engineer.js)는 ERD 엔티티를 선택하고 buildDDL(export.js)로 DDL을 생성한 뒤, 미들웨어 /execute/stream(SSE)을 통해 실제 DB에 적용하는 읽기 전용(ERD 상태 비변경) 기능이다. 모달은 1단계(dialect/옵션) → 2단계(엔티티 선택·충돌 처리·SQL 미리보기·진행률)로 구성되며, 컨텍스트 메뉴(우클릭)에서 단일/다중 엔티티를 지정하여 진입하는 방식도 지원한다. 전반적으로 구조와 비동기/SSE 처리는 견고하나, MSSQL dialect에서 GO 배치 구분자로 인해 실행이 항상 실패하는 치명적 문제와 물리명의 큰따옴표 미이스케이프로 data-pname 속성이 깨지는 문제가 확인됐다. ERD 상태를 변경하지 않으므로 saveState/undo/전체백업 무결성 위반은 없다.

## 2. 발견된 오류 및 수정 결과

### [high] MSSQL DDL의 GO 배치 구분자로 인해 실행이 항상 실패

- **상태**: 수정완료
- **증상**: 대상 DB가 MSSQL일 때 buildDDL이 생성하는 각 CREATE 엔트리에는 SSMS/sqlcmd 전용 배치 구분자인 'GO' 줄이 포함된다(주석 옵션이 꺼져 있어도 CREATE TABLE 뒤에 GO가 붙음). forward_engineer.js는 buildDDL이 반환한 sqls 배열의 각 항목을 그대로 미들웨어 /execute/stream으로 보내고, 미들웨어 mssql 어댑터는 pool.request().query(sql) 한 번에 해당 문자열 전체를 실행한다. mssql 드라이버의 query()는 GO를 이해하지 못하므로 'Incorrect syntax near GO' 등으로 항상 실패한다. 결과적으로 MSSQL dialect의 포워드 엔지니어링이 통째로 동작하지 않는다(진행률은 모든 단계 실패로 표시). 미리보기(text)는 GO 포함이 정상이므로, 실행 시점에만 GO 줄을 분리/제거해 개별 문장으로 보내야 한다.
- **원인**: buildDDL(export.js)이 MSSQL용으로 CREATE/주석 문장 사이에 GO 줄을 넣어 하나의 sqls 항목 안에 다중 배치를 담는데, forward_engineer.js는 이를 분리하지 않고 단일 statement로 전송한다. mssql 어댑터 query()는 GO 미지원이라 구문 오류가 난다.
- **수정 내용**: _feRun()에서 MSSQL용 splitGo() 함수를 추가하여 GO 배치 구분자를 문장 단위로 분리한 뒤 allSqls를 구성하도록 수정, Tedious 드라이버 실행 오류 해소
- **파일**: E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\forward_engineer.js

### [medium] escHtml이 큰따옴표를 이스케이프하지 않아 data-pname 속성이 깨짐(매칭 실패·속성 인젝션)

- **상태**: 수정완료
- **증상**: 엔티티 체크리스트 렌더링 시 물리명을 큰따옴표로 감싼 HTML 속성 data-pname="${escHtml(pname)}"에 삽입한다. 그러나 escHtml(ui.js)은 & < > 만 치환하고 큰따옴표(")는 그대로 둔다. 물리명에 "가 포함되면 속성이 조기 종료되어 el.getAttribute('data-pname')이 잘린 값을 반환하고, 이후 _feUpdateConflictUI/_fePreview/_feRun의 selectedPnames·_feConflicts 키 매칭이 어긋나 충돌 처리(DROP/RENAME/skip)가 오작동한다. 또한 " onmouseover=... 형태의 속성 인젝션(XSS) 가능성도 있다. 같은 패턴이 충돌 UI의 data-pname에도 존재한다(라디오 그룹 키 추출 오류).
- **원인**: escHtml의 치환 집합에 따옴표가 빠져 있어 속성 컨텍스트(쌍따옴표로 감싼 attribute)에 안전하지 않다. 포워드 엔지니어 코드가 escHtml을 속성값 이스케이프 용도로 사용하면서 따옴표 미처리가 그대로 노출된다.
- **수정 내용**: data-pname 속성값 내 큰따옴표를 &quot;으로 이스케이프하여 HTML 속성 파싱 오류 및 속성 인젝션 취약점을 수정
- **파일**: js/forward_engineer.js

## 4. 추천 사항

- **트랜잭션 단위 실행 옵션 제공**: 현재는 SQL을 순차 실행하며 stopOnError로 중단만 가능하다. DROP→CREATE→FK 순서 중간 실패 시 부분 적용으로 스키마가 불일치 상태로 남을 수 있다. 미들웨어 측에서 단일 트랜잭션(BEGIN/COMMIT/ROLLBACK) 실행 모드를 지원하고 클라이언트에서 옵션화하면 원자성이 보장된다(단, DDL 트랜잭션 지원은 DB별 상이 — Oracle은 암묵 커밋).
- **실행 결과 상세 로그/다운로드**: 진행률 외에 각 문장의 성공/실패와 오류 메시지를 누적 표시하고, 실행된 전체 SQL·결과를 텍스트로 복사/다운로드할 수 있게 하면 운영 추적성과 재현성이 향상된다(미들웨어 auditLogger와 연계).
- **RENAME 백업 테이블명 충돌·길이 검증 강화**: _bak_${base36(ts)} 접미사는 동일 ms 내 재실행이나 절단 시 충돌 가능성이 있다. 실행 전 _feExistingTables로 신규 백업명 중복을 검사하고, dialect별 식별자 최대 길이(예: PostgreSQL 63, MSSQL 128)를 정확히 반영하도록 maxLen 테이블을 보강하면 안전하다.
- **미리보기와 실행 SQL의 단일 소스화**: _fePreview는 buildDDL의 text를, _feRun은 sqls를 사용해 두 경로가 분기한다(특히 MSSQL GO 문제처럼). 실행에 사용하는 statement 배열을 그대로 미리보기에도 렌더링하도록 통일하면 '보이는 것과 실행되는 것'의 불일치를 원천 차단할 수 있다.
