# 공유 ▸ P2P 실시간 협업 — 분석 보고서

## 1. 개요

`js/webrtc.js`는 PeerJS WebRTC DataChannel 기반 2자 공동편집을 구현한다. 호스트/게스트 역할 선택, 초대코드 생성·입력, 연결 시 협업 전 자동 스냅샷 저장, 상태 디바운스 전송(saveState 래핑), 수신 시 restoreFromSnapshot 적용, 종료 후 복원 안내, 툴바 배지 주입까지 전체 흐름이 갖춰져 있다. 전반 구조는 양호하나, 수신한 협업 상태를 `saveState()`로 영속화하지 않아 새로고침/undo 시 협업 내용이 유실되는 영속성 무결성 위반이 가장 큰 문제다. 그 외 연결 종료 시 close 이벤트 경쟁과 복원 안내 중복 누적 등 견고성 결함이 있다.

## 2. 발견된 오류 및 수정 결과

### [medium] 수신한 협업 상태를 saveState()로 영속화하지 않아 새로고침/undo 시 유실

- **상태**: 수정완료
- **증상**: 게스트(또는 어느 쪽이든) DataChannel로 상대의 ERD 상태를 수신하면 `restoreFromSnapshot`으로 화면에는 반영되지만 `saveState()`를 호출하지 않는다. 그 결과 (1) localStorage(STORAGE_KEY)에 협업 내용이 저장되지 않아 새로고침하면 협업 전 상태로 되돌아가 협업 결과가 통째로 사라지고, (2) undo 스택에도 push되지 않아 실행취소로도 보존되지 않는다. README의 '데이터 안전 보장' 취지 및 영속성 무결성 규칙(undo/redo 직렬화 대상 누락)에 정면으로 위배된다. `_rtcReceiving=true` 구간 내에서 `saveState`를 호출하면 webrtc 래퍼(라인 368: `!_rtcReceiving` 검사)가 재전송을 건너뛰므로 무한 루프 없이 안전하다.
- **원인**: `_rtcConn.on('data')` 핸들러가 `restoreFromSnapshot`/`render`만 수행하고 영속화 단계인 `saveState()`를 누락. 수신 상태가 휘발성으로만 적용됨.
- **수정 내용**: 수신 데이터 복원 후 `saveState()` 호출을 추가해 협업 상태를 localStorage·undo 스택에 영속화.
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\webrtc.js`

### [low] 연결 종료 시 close() 직후 peer.destroy()로 close 이벤트가 유실되어 복원 안내가 안 뜰 수 있음

- **상태**: 오류아님
- **증상**: `rtcDisconnect()`는 `_rtcConn.close()` 직후 곧바로 `_rtcDestroyPeer()`(=`_peer.destroy()`)를 동기 호출한다. DataConnection의 'close' 이벤트는 비동기로 디스패치되는데, 그 전에 `peer.destroy()`가 연결과 리스너를 정리하면 `_rtcSetupConn`에서 등록한 'close' 핸들러가 실행되지 않아 `_rtcUpdateUI()`와 `_rtcShowRestoreToast()`(복원 안내·배지 해제)가 호출되지 않을 수 있다. 코드 주석 자체가 'close 이벤트가 발생해 자동 호출됨'에 의존한다고 명시하므로, destroy를 close 이벤트 처리 이후로 미루거나 종료 처리를 직접 수행해야 한다.
- **원인**: 비동기 close 이벤트에 의존하는 종료 후처리(UI 갱신·스냅샷 복원 안내)를, 같은 이벤트 루프 틱에서 동기적으로 `peer.destroy()`하여 이벤트가 발화하기 전에 연결/리스너를 파괴함.
- **수정 내용**: 거짓양성으로 판정. 주장의 핵심 전제("DataConnection의 'close' 이벤트가 비동기로 디스패치된다")가 실제 PeerJS 1.5.4 동작과 어긋난다. 검증 결과, 해당 버전에서 `DataConnection.close()`는 동기적으로 `super.emit("close")`를 즉시 발화하며(setTimeout/microtask/promise/브라우저 onclose 콜백을 거치지 않음), eventemitter3 기반이라 emit은 동기로 처리된다. 따라서 실행 순서는 `close()` 호출 → (동기) 'close' 핸들러 실행 → `close()` 반환 → `_rtcDestroyPeer()` 순이며, destroy 이전에 close 후처리가 이미 끝난다. 또한 제안된 수정안을 적용하면 복원 안내 DOM이 중복 append되는 회귀가 발생하므로 수정 불필요.
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\webrtc.js`

### [medium] 종료 후 복원 안내(notice)가 매 종료마다 패널에 중복 누적됨

- **상태**: 수정완료
- **증상**: `_rtcShowRestoreToast()`는 모달이 열려 있으면 `rtcPanelRole`에 복원 안내 div(notice)를 `appendChild`로 매번 새로 추가한다. 기존 notice 제거 로직이 없어, 연결→종료를 반복하거나 모달을 닫지 않은 채 재연결/재종료하면 동일한 '협업 종료' 안내와 복원 버튼이 여러 개 쌓인다. 또한 복원 후에도 이미 추가된 notice가 남을 수 있다.
- **원인**: 안내 div를 멱등(idempotent)하게 다루지 않고 종료마다 새 엘리먼트를 무조건 append. 기존 동일 안내 제거/존재 검사 부재.
- **수정 내용**: 복원 안내 notice에 `id='rtcRestoreNotice'`를 부여하고 생성 전 기존 notice를 제거해 중복 누적 방지.
- **파일**: `E:\04.개발환경\python\98.ETC\SimJaeSugn.github.io\js\webrtc.js`

## 4. 추천 사항

- **협업 세션 식별·표시(피어 닉네임/색상) 및 커서 공유**: 현재는 전체 상태 스냅샷을 통째로 주고받아 마지막 전송이 이기는(last-write-wins) 방식이다. 동시 편집 시 한쪽 변경이 덮어써질 수 있으므로, 최소한 누가 편집 중인지(피어 라벨)와 마지막 동기화 시각을 배지/패널에 표시하면 충돌 인지에 도움이 된다. 장기적으로는 엔티티 단위 부분 패치나 CRDT 도입을 고려할 수 있다.

- **연결 상태/하트비트 기반 끊김 감지 보강**: PeerJS DataConnection의 close/error 이벤트 외에, 주기적 ping/pong 또는 `_peer.on('disconnected')`/reconnect 처리를 추가하면 네트워크 단절을 더 신뢰성 있게 감지·복구할 수 있다. 현재는 상대 탭이 비정상 종료될 때 close가 늦게 오거나 누락될 여지가 있다.

- **전송 페이로드 크기 제한 및 압축**: 대형 ERD(엔티티 수백 개)에서는 매 `saveState`마다 diagrams 전체를 16ms 디바운스로 직렬화·전송하므로 대역폭·CPU 부담이 크다. lz-string은 이미 번들에 있으므로 RTC 전송에도 압축을 적용하거나, 변경된 다이어그램만 전송하는 부분 동기화를 도입하면 효율이 크게 개선된다.

- **PeerJS 클라우드 의존 명시 및 오프라인/실패 폴백 안내**: 시그널링은 PeerJS 공개 클라우드에 의존하므로 해당 서비스 장애 시 연결이 불가능하다. 사용자에게 이 의존성을 README/모달에 안내하고, 연결 실패 시 재시도 버튼과 함께 대안(공유 URL/전체백업 내보내기)으로 유도하면 견고성이 높아진다.
