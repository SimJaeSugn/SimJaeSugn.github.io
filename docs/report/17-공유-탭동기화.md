# 공유 ▸ 탭 동기화 — 분석 보고서

## 1. 개요

`js/broadcast.js`는 BroadcastChannel API를 사용하여 같은 PC 내 여러 탭 간에 ERD 상태를 실시간으로 동기화한다. `saveState()`를 래핑해 16ms 디바운스 후 `{ diagrams, activeDiagramId, viewMode, notationStyle, gridSnap }` 스냅샷을 브로드캐스트하고, 수신 시 `restoreFromSnapshot()`으로 상태를 반영하는 구조다. hello/bye/ping/pong 메시지로 피어를 추적하고 툴바 우측에 '📡 N탭 연결' 배지를 표시하며, `toggleBcSync()`로 동기화를 켜고 끌 수 있다. 자가 메시지 필터, `_bcReceiving` 무한루프 방지, 스테일 메시지 가드 등 핵심 로직은 견고하나, 동기화 인디케이터 페이드 후 배지가 영구히 사라지는 UI 버그와 수신 상태가 localStorage/undo에 반영되지 않아 수동 수신 전용 탭에서 데이터가 분기될 수 있는 견고성 약점이 존재한다.

## 2. 발견된 오류 및 수정 결과

### [medium] 동기화 인디케이터 페이드 후 연결 배지가 영구히 사라짐 (opacity 0 고정)

- **상태**: 수정완료
- **증상**: 다른 탭에서 변경을 수신할 때마다 `_showBcIndicator()`가 호출되어 '📡 동기화됨'을 2초간 보여준 뒤 페이드아웃한다. 페이드 타이머는 `badge.style.opacity`를 `'0'`으로 만든 다음, 0.4초 뒤 `textContent`만 '📡 N탭 연결'로 되돌린다. 그러나 `opacity`를 다시 `'1'`로 복원하지 않으므로, 페이드가 끝난 뒤에는 배지 텍스트는 '📡 N탭 연결'로 바뀌어 있지만 `opacity`가 0이라 화면에서 완전히 안 보인다. 결과적으로 첫 동기화 이후에는 탭이 연결돼 있어도 연결 배지가 사라진 채로 남고, `_updateBcUI()`가 다시 호출되는 이벤트(hello/bye/pong/토글)가 발생하기 전까지 상태 표시가 복구되지 않는다. README가 명시한 '툴바 우측 📡 N탭 연결 배지로 연결 상태 확인' 기능이 깨진다.
- **원인**: `_showBcIndicator`의 중첩 `setTimeout` 콜백이 transient 인디케이터를 영구 라벨로 되돌릴 때 `textContent`만 복원하고 `opacity`를 `'1'`로 되돌리지 않음. `opacity`가 `'0'`으로 남아 배지가 보이지 않음.
- **수정 내용**: `_showBcIndicator`의 페이드아웃 콜백에서 배지 텍스트 복원 후 `opacity`를 `'1'`로 되돌려, 동기화 인디케이터 사라진 뒤 연결 배지가 영구적으로 숨겨지던 버그를 수정
- **파일**: js/broadcast.js

## 3. 개선점 (자동수정 안 함 — 권장)

| 우선순위 | 제목 | 파일 |
|---------|------|------|
| medium | 수신한 동기화 상태가 localStorage/undo에 반영되지 않아 새로고침 시 분기 위험 | js/broadcast.js |
| low | toggleBcSync의 중복/모순되는 display 설정 (미지원 브라우저에서 배지 오표시) | js/broadcast.js |
| low | 탭 종료(bye) 외 비정상 종료/크래시 시 피어 카운트가 영구히 누적 | js/broadcast.js |
| low | 스테일 메시지 가드의 500ms 윈도우가 매직넘버이며 다중 탭 동시 편집 시 손실 가능 | js/broadcast.js |
| low | state 핸들러의 중복 렌더 호출 정리 | js/broadcast.js |

**상세 설명:**

- **수신한 동기화 상태가 localStorage/undo에 반영되지 않아 새로고침 시 분기 위험 (medium)**: `state` 메시지 수신 시 `restoreFromSnapshot(msg.state)`만 호출하고 `saveState()`나 `localStorage.setItem(STORAGE_KEY, ...)`을 하지 않는다. 송신 탭이 열려 있는 동안에는 수신 탭이 새로고침해도 hello→`_bcSendState` 경로로 재동기화되지만, 송신 탭을 닫은 뒤 수신(발표) 탭을 새로고침하면 자신의 오래된 localStorage를 로드해 동기화된 내용이 유실/분기된다. 또한 수신된 변경은 수신 탭의 undo 스택에 없어 되돌릴 수 없다. 수신 후 디바운스로 localStorage에 저장(undo 스택은 선택적으로 제외)하는 것을 검토하면 영속성 무결성이 강화된다.

- **toggleBcSync의 중복/모순되는 display 설정 (low)**: `toggleBcSync()`는 `_updateBcUI()` 호출 직후 다시 `badge.style.display = _bcEnabled ? '' : 'none'`를 설정한다. `_updateBcUI`는 `(_bc && _bcEnabled)`로 `_bc`(채널 생성 실패/미지원 시 null)를 함께 검사하지만, 뒤따르는 중복 라인은 `_bcEnabled`만 본다. 따라서 BroadcastChannel 미지원 브라우저에서 토글 ON 시 동작하지 않는 배지가 보이게 된다. 중복 라인을 제거하고 `_updateBcUI`의 판정에 일임하는 편이 일관적이다.

- **탭 종료(bye) 외 비정상 종료/크래시 시 피어 카운트가 영구히 누적 (low)**: 피어 제거는 `beforeunload`의 `_bcPost({type:'bye'})`에만 의존한다. 탭 크래시·강제 종료·`beforeunload` 미발화 시 `_bcPeers`에서 제거되지 않아 'N탭 연결' 수가 실제보다 크게 남는다. 주기적 ping 후 일정 시간 내 pong 미수신 피어를 만료(heartbeat/TTL)하는 메커니즘을 추가하면 카운트 정확도가 향상된다.

- **스테일 메시지 가드의 500ms 윈도우가 매직넘버이며 다중 탭 동시 편집 시 손실 가능 (low)**: `state` 핸들러의 `'if (msg.ts < _bcLastSent - 500) break;'`는 자기 송신보다 0.5초 이상 오래된 원격 상태를 폐기한다. 시계 기준이 각 탭 `Date.now()`로 동일 PC라 보통 안전하지만, 두 탭이 거의 동시에 편집하면 한쪽 변경이 조용히 무시될 수 있다. 상수를 명명 상수로 추출하고, 동시 편집 충돌 정책(마지막 쓰기 우선 등)을 문서화하는 것을 권장한다.

- **state 핸들러의 중복 렌더 호출 정리 (low)**: `state` 수신 시 `restoreFromSnapshot(msg.state)` 내부에서 이미 `syncToolDropdownLabels()`, `setViewMode(viewMode)`, `updateZoomLabel()`, `renderDiagramPanel()`, `render()`를 호출하는데, 핸들러에서 `renderDiagramPanel?.()`, `updateZoomLabel?.()`, `setViewMode?.(viewMode)`, `syncToolDropdownLabels?.()`, `render()`를 한 번 더 호출한다. 기능상 무해하나 매 수신마다 중복 렌더가 발생해 성능 낭비다. `restoreFromSnapshot`에 일임하고 중복 호출을 제거하는 편이 좋다.

## 4. 추천 사항

- **단축키 동기화 점검 결과: 탭 동기화 토글에 단축키 없음 (현 상태 일관)**: `toggleBcSync`는 `main.js`/`shortcuts.js`/단축키 모달 어디에도 바인딩이 없고 `index.html` 공유 메뉴와 배지 클릭으로만 접근한다. 충돌·불일치는 없다. 자주 토글한다면 `matchSC` 기반 단축키를 추가하고 `shortcutsOverlay` 모달에도 함께 등록하는 것을 고려할 수 있다.

- **`_bcEnabled` 토글 상태를 localStorage에 영속화**: 현재 `_bcEnabled`는 항상 `true`로 초기화되어 새로고침하면 동기화가 다시 켜진다. 사용자가 끈 상태를 기억하도록 localStorage에 저장/복원하면 발표용 탭 등에서 의도치 않은 재동기화를 막을 수 있다. (이 값은 영속 ERD 데이터가 아니므로 전체백업/undo 직렬화 대상에는 넣지 않는 것이 적절하다.)

- **수신 시 사용자 확인 또는 시각적 디프 옵션**: 현재는 수신 즉시 무조건 `restoreFromSnapshot`으로 현재 작업 내용을 덮어쓴다. 동시 편집 시 진행 중이던 로컬 변경이 통째로 교체될 수 있으므로, 수신 적용 전 토스트로 알리거나 '적용/무시' 선택을 주는 옵션을 두면 데이터 손실 위험을 줄일 수 있다.
