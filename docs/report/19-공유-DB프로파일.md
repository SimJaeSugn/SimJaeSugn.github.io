# 공유 ▸ DB 프로파일 관리 — 분석 보고서

## 1. 개요

'공유 ▸ DB 프로파일 관리' 메뉴는 `js/profile_manager.js`(모달 UI·CRUD)와 `js/db_connect.js`(미들웨어 핑/설정 조회)로 구성되며, MW_URL(127.0.0.1:3737)의 `/config/profiles` 계열 REST API와 통신한다. 모달은 1회만 생성하고 재호출 시 목록만 갱신하는 구조로 이벤트 리스너 누수는 없고, 비동기 fetch는 모두 try/catch로 감싸 미들웨어 미실행·네트워크 오류를 처리한다. 이 메뉴는 전역 상태(diagrams/ENTITIES 등)나 localStorage·undo/redo·전체백업에 영속되는 데이터를 다루지 않으므로(프로파일은 미들웨어 측에 저장) 영속성 무결성·단축키 동기화 위반은 발견되지 않았다. 다만 사용자 입력을 인라인 onclick 핸들러의 JS 문자열 리터럴에 삽입할 때 `_pmEsc`(HTML 엔티티 인코딩)만 사용해, 작은따옴표가 포함된 이름에서 핸들러가 깨지거나 스크립트가 주입되는 실제 버그가 존재한다.

## 2. 발견된 오류 및 수정 결과

> **✅ 후속 수정 완료 (feature-dev 파이프라인):** 아래 5개 항목은 모두 동일한 근본 원인 — `_pmEsc`(HTML 엔티티 인코더)를 `onclick` 의 JS 문자열 리터럴 컨텍스트에 사용 — 에서 비롯된다. `js/profile_manager.js` 에 JS-문자열 + double-quoted HTML 속성 양쪽에 안전한 `_pmEscJsAttr()` 헬퍼를 신규 정의하고, 전환·편집(6개 인자)·삭제·선택·저장 버튼의 onclick 인자를 이 헬퍼 기반 변수(`jName` 등)로 교체하여 **모두 수정 완료**했다(표시용 innerHTML 텍스트는 `_pmEsc` 유지). 자동 워크플로 검증 단계에서 보류됐던 사유(원 제안 패치가 미정의 헬퍼 `_pmEscJsAttr`/변수 `eJs`·`eNameJs` 를 참조 → ReferenceError)는, 헬퍼와 참조 변수를 실제로 정의해 자기완결적으로 구현함으로써 해소했다. `node --check` 통과, reviewer PASS. 아래 각 항목의 **'수정 내용'** 은 *최초 보류 사유 기록*이다.

### [high] 인라인 onclick 인자에 작은따옴표 포함 이름 삽입 시 핸들러 깨짐/스크립트 주입

- **상태**: 수정완료 *(feature-dev 후속 수정)*
- **증상**: 프로파일 이름·호스트·DB명 등에 작은따옴표(')가 포함되면 전환/편집/삭제/선택 버튼의 onclick이 조용히 실패한다(JS 구문 오류). 예: 이름이 O'Brien이면 `_pmEsc`로 `O&#39;Brien`이 되지만, 브라우저가 HTML 속성을 파싱할 때 `&#39;`가 다시 `'`로 디코딩되어 실제 실행 코드는 `_pmSelectProfile('O'Brien')`가 되어 SyntaxError가 발생한다. 또한 `'),someFn('` 형태의 이름이면 onclick 컨텍스트에서 임의 스크립트가 주입될 수 있다(저장 XSS). 추가 폼은 이름에 임의 문자열을 허용(.trim()만)하고 미들웨어도 비어있지 않은 이름이면 모두 수용하므로 충분히 재현 가능하다.
- **원인**: `_pmEsc()`는 텍스트(innerHTML) 컨텍스트용 HTML 엔티티 인코더이며 `'`를 `&#39;`로 바꾼다. 그러나 `onclick="_pmSelectProfile('...')"` 처럼 '이중따옴표 HTML 속성 안의 작은따옴표 JS 문자열 리터럴' 컨텍스트에서는 `&#39;`가 속성 파싱 단계에서 다시 `'`로 환원되어 JS 문자열을 탈출시킨다. 이 컨텍스트에서는 작은따옴표를 백슬래시(`\'`)로 이스케이프해야 하는데, 전용 헬퍼 없이 `_pmEsc`만 사용한 것이 근본 원인이다. 코드베이스의 다른 패널(diagrams.js 등)은 dataset.id를 쓰는 안전한 패턴을 사용하지만 이 모듈만 문자열을 인라인 핸들러에 직접 끼워넣는다.
- **수정 내용**: 버그 자체는 실재하나(검증 완료), 올바른 수정은 JS-속성 컨텍스트의 모든 호출부를 `_pmEscJsAttr`로 교체해야 하며, 이는 `_renderProfileList`의 map 본문(line 295-333)과 `_renderRightPanel`의 edit 분기(line 476/535)에 걸쳐 있다. 단일 anchor 교체로 끝나지 않고 헬퍼 추가 + 다수 호출부 변경 + 컨텍스트별 변수 분리가 필요해 회귀 검토가 필요하다. 자동수정 제외, 다중 지점 변경으로 수동 적용 권장.

### [medium] 전환/편집/삭제/선택 onclick 인자에 안전하지 않은 _pmEsc 사용

- **상태**: 수정완료 *(feature-dev 후속 수정)*
- **증상**: `_renderProfileList`에서 전환(`_activateProfile`)·편집(`_openEditProfileForm`)·삭제(`_deleteProfile`)·선택(`_pmSelectProfile`) 버튼의 onclick 인자를 `_pmEsc`로만 처리한다. 작은따옴표가 포함된 프로파일 이름/호스트/DB명에서 해당 버튼들이 동작하지 않거나 스크립트 주입이 발생한다. editArgs는 name/dbType/host/database/username/clientLibDir를 모두 JS 문자열 인자로 넘기므로 이들 중 하나라도 작은따옴표를 포함하면 편집 버튼 전체가 깨진다.
- **원인**: 인라인 핸들러의 JS 문자열 리터럴 컨텍스트인데 텍스트용 `_pmEsc`를 사용. e1의 헬퍼(`_pmEscJsAttr`)로 교체해야 한다. (목록 표시용 텍스트 `${eName}`, info 영역은 텍스트 컨텍스트이므로 `_pmEsc` 유지가 맞다.)
- **수정 내용**: 버그 자체는 실재하나, 제안된 replacement는 `_pmEscJsAttr` 헬퍼가 코드베이스 어디에도 정의돼 있지 않아 그대로 적용하면 ReferenceError로 정상 프로파일 포함 전체 목록 렌더링이 깨진다. 올바른 수정은 (a) `_pmEscJsAttr` 헬퍼를 `_pmEsc` 정의 근처에 추가하고, (b) 인라인 onclick 인자 컨텍스트에서만 교체하는 2단계 작업이 필요하다. 자동수정 제외, 헬퍼 추가를 포함한 수정으로 별도 진행 권장.

## 4. 추천 사항

- **인라인 onclick 대신 데이터셋 + 이벤트 위임 패턴으로 통일**: `diagrams.js`의 `dataset.id` 패턴처럼 프로파일 항목/버튼에 `data-name`을 부여하고 목록 컨테이너에 단일 click 위임 핸들러를 등록하면, 사용자 입력을 onclick 문자열에 끼워넣을 필요가 없어져 따옴표/특수문자 이스케이프 버그 클래스를 근본적으로 제거할 수 있다.

- **프로파일 추가 폼에 연결 테스트(Test Connection) 버튼 추가**: `db_connect.js`에는 미들웨어 핑(`_mwPing`)만 있고, 저장 전 실제 DB 접속을 검증하는 흐름이 UI에 없다. 미들웨어에는 `/config/test`(POST) 엔드포인트가 존재하므로, 추가/편집 폼에 '연결 테스트' 버튼을 두면 잘못된 접속정보 저장을 사전에 방지할 수 있다.

- **이름 입력 검증 규칙 명문화**: 프로파일 이름에 허용할 문자 집합(예: 따옴표·꺾쇠 금지 또는 길이 제한)을 추가 폼에서 검증하면 UI 깨짐과 주입 위험을 입력 단계에서 차단할 수 있다. 서버(POST/PUT)에서도 동일 검증을 적용하면 클라이언트 우회에도 안전하다.
