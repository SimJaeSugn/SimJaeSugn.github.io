# 다이어그램 ▸ 패널·트리·속성 — 분석 보고서

## 1. 개요

'다이어그램 ▸ 패널·트리·속성' 메뉴는 diagrams.js(패널/트리/탭 색상/리사이즈/diff)와 ui.js(우측 속성 패널 propPanel, 컨텍스트 메뉴)로 구성되며, 다이어그램 목록 CRUD·전환, 엔티티 트리(renderEntityTree), 엔티티/관계 인라인 속성 편집, 다이어그램 간 복사(confirmCopyToDiag)를 제공한다. saveState()/flushCurrentState() 플러시와 undo/redo 직렬화, localStorage 저장 흐름은 전반적으로 일관되게 잘 연결되어 있으며 onclick 핸들러도 모두 실제 전역 함수와 매칭된다. 다만 엔티티 트리의 PK/FK 배지가 존재하지 않는 필드(a.pk/a.fk)를 참조해 항상 표시되지 않는 명백한 버그와, 다이어그램 전환/생성 시 이전 선택(selectedEntity)과 열린 속성 패널(_ppTarget)이 초기화되지 않아 분리(detached)된 객체를 편집·강조하게 되는 상태 불일치 버그가 확인되어 수정 완료되었다. 데이터 모델은 attr.kind('pk'|'fk'|'normal')를 단일 진실 소스로 사용한다.

## 2. 발견된 오류 및 수정 결과

### [medium] 엔티티 트리 PK/FK 배지가 존재하지 않는 필드(a.pk/a.fk)를 읽어 항상 미표시

- **상태**: 수정완료
- **증상**: 우측 패널의 엔티티 트리(renderEntityTree)에서 속성 펼치기 시 각 속성의 PK/FK 배지를 a.pk / a.fk 값으로 결정한다. 그러나 이 코드베이스의 속성 데이터 모델은 종류를 attr.kind('pk'|'fk'|'normal') 하나로만 표현한다(canvas.js의 렌더링 988~1021행, entities.js의 FK 정렬·자동생성 212~614행, ui.js 속성 패널 _buildAttrRow 1295~1306행 모두 attr.kind 사용). a.pk / a.fk 라는 불리언 필드는 코드 어디에서도 설정되지 않으므로 항상 undefined(falsy)다. 결과적으로 트리에서는 PK·FK 속성이라도 항상 'tree-badge-normal'(빈 텍스트) 배지로만 렌더되어 README 75·110·111행이 명시한 'PK/FK 표시' 사양과 어긋난다. 재현: 임의 엔티티의 속성을 PK 또는 FK로 지정한 뒤 우측 트리에서 해당 엔티티를 펼치면 PK/FK 배지가 나타나지 않는다.
- **원인**: 속성 종류의 단일 진실 소스가 attr.kind인데 renderEntityTree만 구버전/타 모델의 a.pk·a.fk 불리언을 참조한다. 두 곳의 모델 표현이 어긋난 미동기화 버그.
- **수정 내용**: 엔티티 트리 PK/FK 배지 조건을 a.pk/a.fk에서 a.kind === 'pk'/'fk' 비교로 수정하여 배지가 올바르게 표시되도록 함
- **파일**: js/diagrams.js

### [high] 다이어그램 전환 시 이전 선택/속성 패널이 분리된 객체를 가리켜 편집이 유실됨

- **상태**: 수정완료
- **증상**: switchDiagram()은 flushCurrentState()로 현재 작업을 저장하고 loadDiagramIntoWorkspace()로 ENTITIES를 새 객체로 교체하지만, selectedEntity·selectedEntities와 우측 속성 패널 상태(_ppTarget)를 초기화하지 않는다. 따라서 (a) 이전 다이어그램의 엔티티가 selectedEntity로 남아 render()에서 새 다이어그램 위에 잘못된 강조가 나타날 수 있고, (b) 속성 패널이 열려 있었다면 _ppTarget이 이전 다이어그램의(이제 어떤 diagrams[]에도 속하지 않는) 분리된 엔티티 객체를 계속 가리킨다. 이 상태에서 패널의 논리명/물리명/컬럼을 수정하면 saveState→flushCurrentState 시 분리 객체는 직렬화 대상에서 제외되어 변경이 조용히 사라진다(데이터 유실). 재현: A 다이어그램의 엔티티를 클릭해 속성 패널을 연 뒤 다른 다이어그램 탭으로 전환하면 패널이 여전히 A의 엔티티를 표시하고, 거기서 편집한 내용은 저장되지 않는다.
- **원인**: 다이어그램 전환 시 작업 배열(ENTITIES 등)은 새 클론으로 교체되는데, 그 배열을 참조하던 UI 선택 상태(selectedEntity/_ppTarget)는 갱신·초기화되지 않아 stale 참조가 남는다.
- **수정 내용**: switchDiagram 함수에서 다이어그램 전환 시 selectedEntity 초기화, selectedEntities.clear(), hidePropPanel 호출을 추가하여 stale 선택/속성 패널 문제 해결
- **파일**: js/diagrams.js

### [medium] 새 다이어그램 생성 시 이전 선택/속성 패널 미초기화로 stale 편집 발생

- **상태**: 수정완료
- **증상**: confirmNewDiag()도 switchDiagram과 동일하게 flushCurrentState 후 새 빈 다이어그램을 활성화하고 loadDiagramIntoWorkspace로 작업 배열을 교체하지만, selectedEntity·selectedEntities·속성 패널(_ppTarget)을 초기화하지 않는다. 직전 다이어그램의 엔티티를 선택/속성 패널로 편집 중에 '＋'(새 다이어그램)을 누르면, 빈 다이어그램이 표시되는데도 우측 속성 패널은 이전 엔티티(분리 객체)를 계속 가리키며 거기서의 편집은 저장되지 않는다.
- **원인**: switchDiagram과 동일 — 작업 배열 교체 시 UI 선택/패널 stale 참조 미정리.
- **수정 내용**: confirmNewDiag 함수에서 새 다이어그램 생성 시 selectedEntity 초기화, selectedEntities.clear(), hidePropPanel 호출을 추가하여 이전 선택 상태 미초기화 문제 해결
- **파일**: js/diagrams.js

## 4. 추천 사항

- **속성 모델 단일화 자기검증 추가**: attr.kind를 단일 진실 소스로 강제하기 위해 migrateAttr 단계에서 레거시 a.pk/a.fk가 있으면 kind로 흡수하고 제거하는 정규화를 넣고, 렌더 경로에서 kind만 참조하도록 통일하라. e1 같은 모델 미동기화 버그의 재발을 구조적으로 막는다.
- **다이어그램 전환/생성/삭제 공통 진입점 도입**: switchDiagram·confirmNewDiag·deleteDiagram·restoreFromSnapshot이 공통으로 수행해야 하는 'UI 선택 상태 초기화(selectedEntity/selectedEntities/_ppTarget/focusEntityId)'를 한 함수(resetSelectionState 등)로 모아 호출하면 stale 참조 버그를 일괄 예방할 수 있다.
- **속성 패널-캔버스 양방향 동기화 강화**: 엔티티가 다른 경로(컨텍스트 메뉴 편집, 가져오기, 협업 수신 등)로 변경될 때 열린 속성 패널을 자동 갱신하는 refreshPropPanel 호출 지점을 표준화하면 화면-상태 일치를 보장할 수 있다.
