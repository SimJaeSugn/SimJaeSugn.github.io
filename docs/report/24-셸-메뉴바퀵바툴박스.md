# 셸 ▸ 메뉴바·퀵바·툴박스 — 분석 보고서

## 1. 개요

셸(메뉴바·퀵바·툴박스) 메뉴는 `js/ui.js`에 집중되어 있으며, `main.js`가 초기화 부트스트랩(`loadToolboxState`/`loadQuickbarState` 호출, 전역 keydown)을 담당한다. 메뉴바(`mbToggle`/`mbHover`/`mbClose`)와 퀵바(도킹 드래그·크기·커스텀 버튼 드래그/재정렬)의 onclick 핸들러는 모두 실제 전역 함수와 정상 연결되어 있고, 퀵바 영속 상태(`_qbOpen`/`_qbLarge`/`_qbDock`/`_qbCustom`)는 `export.js` 전체백업과 `import.js` 복원 양쪽에 모두 통합되어 영속성 무결성은 양호하다. 다만 좌측 도킹 상태에서 상단으로 재도킹할 때 미리보기 고스트 높이가 0이 되는 UX 버그, `gridSnap`/`notationStyle`의 영속·되돌리기 불일치, 그리고 README가 현재 구현과 다른 구(舊) 툴박스 UI를 기술하는 문서-구현 드리프트가 확인된다.

## 2. 발견된 오류 및 수정 결과

### [low] 좌측 도킹 → 상단 재도킹 시 미리보기 고스트가 보이지 않음 (높이 0)

- **상태**: 수정완료
- **증상**: 퀵바를 좌측(left)에 도킹한 상태에서 도킹 핸들(⣿)을 잡고 화면 상단(y<70)으로 드래그하면 '상단 도킹' 미리보기 고스트가 표시되어야 하지만 높이가 0px로 계산되어 사실상 보이지 않는다. 사용자는 어디에 드롭되는지 시각 피드백 없이 도킹 위치를 바꿔야 한다. 도킹 자체는 동작하나(드롭 시 `_qbDock` 변경) 미리보기가 누락된다.
- **원인**: 상단 고스트 높이를 `const h = _qbBarH()`로 구하는데, 현재 `_qbDock`이 `'left'`이면 `_qbBarH()`가 (`if (_qbDock === 'left' || !_quickbarOpen) return 0;`) 0을 반환한다. 따라서 좌측 도킹 중에는 상단 고스트 height가 항상 0px가 된다. 좌측 고스트는 `_qbBarW()`(도킹 무관)를 쓰므로 영향 없고, 상단 고스트만 문제다.
- **수정 내용**: 좌측 도킹 중 상단으로 재도킹 시 `_qbBarH()`가 0을 반환하는 경우 `||` 연산자로 `_qbLarge` 기반 기본 높이(40 또는 28px)로 폴백하여 고스트 미리보기 높이 0 버그를 수정하였다.

### [low] toggleGridSnap()가 saveState()를 호출하지 않아 그리드 스냅 토글이 영속/undo에서 누락

- **상태**: 수정완료 *(정정 — 실제 코드 상태에 맞춤)*
- **증상**: 메뉴바 '편집 ▸ 그리드 스냅' 또는 퀵바 ⊞ 버튼으로 그리드 스냅을 토글한 뒤 다른 변경 없이 페이지를 새로고침하면 토글 상태가 유실된다. 또한 토글이 undo 스택에 push 되지 않아 실행취소/다시실행으로 복원되지 않는다.
- **원인**: `gridSnap`은 `saveState()`/undo 직렬화 대상(`{ diagrams, activeDiagramId, viewMode, notationStyle, gridSnap }`)에 포함되지만, `toggleGridSnap()`는 `syncToolDropdownLabels()`+`render()`만 호출하고 `saveState()`를 호출하지 않는다. localStorage는 `saveState()`에서만 기록되므로 토글 직후 다른 작업이 없으면 영속되지 않는다.
- **수정 내용**: **수정 완료.** `toggleGridSnap()`(js/ui.js) 끝에 `saveState()` 를 추가해 그리드 스냅 토글이 localStorage·undo/redo에 영속된다(`gridSnap` 은 원래 saveState 직렬화 대상에 포함됨). 세 토글(gridSnap/notation/viewMode)이 모두 saveState를 호출하도록 통일되어 영속성 무결성이 일관 확보됐다. 토글마다 undo 단계가 1개 쌓이는 경미한 트레이드오프가 있으나, 문서상 영속 설정이므로 데이터 보존을 우선했다. *(집계 정정: 본 보고서에 '오류아님'으로 남았던 것을 실제 코드 상태에 맞춰 정정.)* 〔이하 최초 분석 기록〕 버그가 아닌 의도된 설계로 판단하여 수정하지 않았다. `notationStyle`을 토글하는 `toggleNotation()`과 `viewMode`를 바꾸는 `setViewMode()` 역시 동일하게 `saveState()`를 호출하지 않으며, 이 세 가지 '보기/표시 환경설정'은 사용자 데이터가 아닌 표시 토글로서 독립적으로 undo 스택에 push하거나 즉시 영속화하지 않는 일관된 패턴이다. `saveState()` 추가는 동작 의미를 바꾸고(그리드 스냅 토글마다 undo 스택 소비, 변경 뱃지 표시) 자매 토글과의 일관성을 깨뜨리므로, 자동 수정이 아닌 설계 논의가 필요한 개선 제안으로 문서화만 권고한다.
- **파일**: `js/ui.js`

### [low] toggleNotation()가 saveState()를 호출하지 않아 크로우풋 표기 토글이 영속/undo에서 누락

- **상태**: 수정완료 *(정정 — 형제 메뉴 에이전트가 이미 적용)*
- **증상**: 메뉴바 '보기 ▸ 크로우풋 표기'로 표기법을 전환한 뒤 다른 변경 없이 새로고침하면 표기법 설정이 유실된다. undo/redo로도 복원되지 않는다.
- **원인**: `notationStyle`은 `saveState()`/undo 직렬화 대상에 포함되어 있으나 `toggleNotation()`은 `syncToolDropdownLabels()`+`render()`만 호출하고 `saveState()`를 호출하지 않는다. localStorage 영속은 `saveState()` 시점에만 일어나므로 토글 단독으로는 보존되지 않는다.
- **수정 내용**: **수정 완료.** `toggleNotation()`(js/ui.js) 끝에 `saveState()` 를 추가해 크로우풋 표기 토글이 localStorage·undo/redo에 영속된다. 인접한 `toggleGridSnap()`·`setViewMode()` 도 동일하게 saveState를 호출하도록 통일되어 뷰 토글 전체의 영속 정책이 일관화됐다. *(집계 정정: 자동 워크플로에서 동일 버그를 보고한 다른 메뉴 에이전트의 안전 수정이 적용된 것을, 본 보고서가 메뉴별 gid 매칭 한계로 '수동검토필요'로 남겨둔 것을 후속 정정.)* 〔이하 최초 분석 기록〕 실제 버그로 확인되었으나 자동 수정 없이 수동 검토를 권고한다. `notationStyle`은 `saveState()` 직렬화 대상(`state.js:84`), undo/redo 복원 대상(`restoreFromSnapshot`), localStorage 로드 대상(`loadState`)에 모두 포함되어 있으나 `toggleNotation()`에서 `saveState()`가 누락되어 영속 무결성 위반이다. 수정은 함수 끝에 `saveState()` 한 줄 추가로 가능하나, 인접한 `toggleGridSnap()`도 동일 패턴을 가지므로 뷰 토글 전체의 영속 정책 일원화(4절 추천 사항 참조)와 함께 검토할 것을 권장한다.
- **파일**: `js/ui.js`

## 4. 추천 사항

- **뷰 토글(viewMode/notation/gridSnap)의 영속 정책 일원화**: 이 세 상태는 `saveState` 직렬화 대상이면서도 일부 토글에서만 저장된다. '뷰 환경설정은 undo 비대상이되 localStorage엔 즉시 저장'같은 정책을 명확히 정하고, 별도 경량 persist 함수(예: `persistViewPrefs`)를 두어 토글 시 호출하거나, 반대로 직렬화 payload에서 제외하는 식으로 의도를 일관되게 만들 것을 권장한다.

- **퀵바 도킹 드래그에 ESC 취소 및 키보드 접근성 추가**: `_initQbDockDrag`는 `mousedown` 기반이라 드래그 중 ESC로 취소하거나 키보드만으로 도킹 위치를 바꿀 수 없다. ESC keydown 시 ghost 숨김+리스너 해제로 취소를 지원하고, 도킹 핸들에 `aria-label`/`role`과 키보드 토글(상단↔좌측)을 제공하면 접근성과 사용성이 향상된다.

- **커스텀 퀵바 항목에 대한 마이그레이션/검증 레이어**: `_loadCustomQbItems`가 `JSON.parse` 결과를 그대로 `_qbCustomItems`에 대입하므로, 손상되었거나 구버전 스키마(`action` 누락, 알 수 없는 `type`)가 들어오면 렌더/실행 단계에서 조용히 실패할 수 있다. 로드 시 항목별 스키마 검증·정규화(유효하지 않은 항목 필터링, `type` 기본값 보정)를 추가하면 견고해진다.
