# 설정 ▸ 기본데이터·상수 — 분석 보고서

## 1. 개요

`js/config.js`는 새 다이어그램의 기본 엔티티/관계, 스타일 상수(W·ROW_H 등), 4개 테마 색상표(THEMES)와 베이스 COLOR 객체, 엔티티/섹션/메모 색상 팔레트, GRID·UNDO_MAX·각종 localStorage 키, DB 타입별 자료형 목록(DB_TYPES), 컬럼 템플릿 기본값(DEFAULT_TEMPLATES)을 정의하는 순수 상수/기본데이터 모듈이다. 대부분의 상수는 일관성 있게 정의되어 있으며, DB_TYPES·DEFAULT_TEMPLATES·DEFAULT_ENTITIES의 FK는 DEFAULT_RELATIONS와 정합하고, 영속성 키들(TEMPLATE_KEY·AI_KEY_STORAGE·THEME_STORAGE·SNAPSHOT_KEY)은 전체 백업 export/import에 모두 포함되어 영속성 무결성 규칙을 만족한다. 다만 베이스 COLOR 객체가 THEMES.dark가 정의하는 `ac_y`·`sectionLabelBg` 두 색상 키를 누락하고 있어, 저장된 테마가 없는 첫 로드 시 canvas 렌더에서 해당 색이 `undefined`가 되는 실제 시각 버그가 존재한다. 일부 기본 관계의 카디널리티 의미가 데이터 모델상 어색한 점은 개선 후보로 남아 있다.

## 2. 발견된 오류 및 수정 결과

### [medium] 베이스 COLOR 객체에 ac_y·sectionLabelBg 누락 → 저장 테마 없는 첫 로드 시 렌더 색상 undefined

- **상태**: 수정완료
- **증상**: 새 사용자/시크릿창 등 localStorage에 `erd_theme`가 없는 첫 로드에서는 `main.js`의 `loadSavedTheme()`가 '저장된 테마가 있을 때만' `applyTheme()`를 호출한다(`ui.js: if (saved && THEMES[saved]) applyTheme(saved, false);`). 따라서 `applyTheme` 내부의 `Object.assign(COLOR, theme.color)`가 실행되지 않고, COLOR 객체는 `config.js`에 정의된 베이스 키만 갖는다. 그런데 베이스 COLOR(라인 126~150)에는 `ac_y`와 `sectionLabelBg`가 없고 이 두 키는 THEMES.dark에만 들어 있다. `canvas.js`는 렌더 시 `COLOR.ac_y`(볼륨 배지/하이라이트, 라인 877·878·941·946·957·1005)와 `COLOR.sectionLabelBg`(섹션 라벨 배경, 라인 1611)를 직접 사용한다. 결과적으로 첫 렌더에서 `ctx.fillStyle = COLOR.sectionLabelBg`가 `undefined`로 설정되어(캔버스는 invalid fillStyle을 무시하므로) 섹션 라벨 배경이 직전 색을 그대로 쓰거나, 데이터량 볼륨 배지 색(`ac_y`)이 의도와 다르게 칠해진다. 테마 모달을 한 번 열어 다른 테마를 고르거나(=`applyTheme` 실행) 페이지를 재방문(테마 저장 이후)하면 증상이 사라지므로 재현이 헷갈릴 수 있다.
- **원인**: 베이스 COLOR 상수(첫 렌더 시 적용되는 기본값)가 THEMES.dark.color의 색상 키 집합과 어긋나 있음. dark 테마는 `ac_y`·`sectionLabelBg`를 추가로 정의하지만 베이스 COLOR에는 두 키가 빠져 있어, `applyTheme`가 호출되기 전(저장 테마 미존재 시) 두 색이 `undefined`로 남는다.
- **수정 내용**: COLOR 객체(베이스)에 `ac_y`·`sectionLabelBg` 키를 추가 — 앞선 수정(`canvas-sections:e1`과 동일 위치)으로 이미 반영되어 중복 적용 없이 완료.
- **파일**: `js/config.js`

## 4. 추천 사항

- **테마 색상 키 정합성 자동 검증**: 애플리케이션 부팅 시(개발 모드) 베이스 COLOR의 키 집합과 THEMES 각 테마 color의 키 집합을 비교해 누락/추가 키를 `console.warn`으로 알리는 가벼운 가드를 추가하면, 이번 `ac_y`/`sectionLabelBg` 누락 같은 비동기화를 조기에 발견할 수 있다.
- **DB_TYPES와 기본 타입 사전 점검 유틸 제공**: `DEFAULT_ENTITIES`에 쓰인 자료형(VARCHAR, INTEGER 등)이 선택된 DB의 DB_TYPES 목록에 없을 때 `buildTypeOptions`가 임시로 맨 위에 끼워넣는 방식인데, 기본 데이터의 타입을 각 DB_TYPES에 모두 포함시키거나 '표준 타입 → DB별 매핑' 테이블을 두면 DB 전환 시 타입 표시가 더 일관된다.
- **localStorage 키 버저닝/마이그레이션 일원화**: `STORAGE_KEY='uxerd_v3'`처럼 버전 접미사를 쓰는 키들이 여러 개(SNAPSHOT_KEY, TEMPLATE_KEY, AI_KEY_STORAGE, THEME_STORAGE) 흩어져 있다. 키 목록과 버전·마이그레이션 규칙을 `config.js` 한 곳에 모아 상수로 노출하면 백업/복원·스키마 변경 시 누락 위험을 줄일 수 있다.
